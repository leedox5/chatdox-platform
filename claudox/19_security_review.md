# 19. 보안 리뷰 체크리스트

보안 문제는 비밀번호와 API 키에서만 생기는 것이 아니었다. 프로젝트가 문서 중심으로 운영되자 이름, 대화 기록, 내부 기획, 요청 티켓과 커밋 메시지도 외부로 나갈 수 있는 데이터가 됐다.

LEEDOX 팀은 GitHub push 과정에서 이 사실을 직접 경험했다. 개인·프로젝트 맥락을 담은 새 문서를 외부 Remote로 보내려 하자 실행 환경의 보안 검토가 명령을 차단했다. 원본 문서를 제외한 뒤에도, 그 내용을 요약한 서비스데스크 기록이 간접 노출로 판정됐다.

이 사건은 귀찮은 장애물이 아니라 문서 보안의 경계를 다시 정하는 계기가 됐다.

## 외부 전송은 별도의 행위다

파일을 로컬에서 읽고 수정하는 것, Git에 커밋하는 것, Remote로 push하는 것은 서로 다른 단계다.

| 작업 | 외부 전송 |
|---|:---:|
| 파일 읽기·수정 | 아니요 |
| 테스트와 diff 확인 | 아니요 |
| 로컬 Git 커밋 | 아니요 |
| Git push | 예 |
| 외부 API·업로드·메시지 | 예 |

사용자가 요청했다는 사실만으로 모든 외부 전송이 허용되는 것도 아니었다. 실행 환경의 상위 보안 정책이 전송 대상과 내용을 별도로 판단할 수 있었다. 한 번 거부된 결과를 다른 명령으로 우회해서는 안 됐다.

## 원본만 가리면 충분하지 않다

민감한 문서를 제외해도 티켓, 대시보드, 로그와 커밋 메시지에 같은 내용이 요약되어 있을 수 있었다.

외부 전송 전에는 다음을 함께 확인해야 했다.

- staged 파일 목록과 실제 diff
- 새로 생성된 문서
- 서비스데스크 요청과 작업 로그
- 자동 생성된 대시보드와 색인
- 커밋 메시지
- 파일명과 문서 링크
- 이전 커밋에 이미 들어간 정보
- Remote URL과 저장소 공개 범위

Private 저장소도 로컬 전용은 아니었다. 접근할 수 있는 사람이 제한될 뿐 외부 서비스로 전송된다는 사실은 같았다.

## 로컬 기억과 공식 지식을 분리하다

개인적인 관계와 세션 기억은 CONTEXT.local.md와 .local/chatdox 아래로 이동했다. 완료된 handoff 원본도 .local/handoff/completed에 보관했다.

반면 다른 팀원과 다른 PC에서도 필요한 일반 원칙은 공식 문서로 정제했다.

- 가상의 AI 팀 운영 모델
- LEEDOX 홈 협업 사례
- Git 업무 문서 관리 기준

이렇게 하면 원본의 민감한 맥락을 모두 공개하지 않고도, 협업에서 얻은 지식은 팀 자산으로 남길 수 있었다.

## 보안 리뷰 체크리스트

### 비밀정보

- [ ] API 키, 토큰과 비밀번호가 없는가
- [ ] 인증서, 개인키와 세션 정보가 없는가
- [ ] 결제수단과 고객 민감정보가 없는가
- [ ] 환경 변수 파일이 포함되지 않았는가

### 개인·업무 맥락

- [ ] 개인 이름과 연락처의 공개가 승인됐는가
- [ ] 대화 원문 대신 필요한 결정만 요약했는가
- [ ] 내부 가격·상품 전략이 불필요하게 노출되지 않는가
- [ ] 티켓과 로그에 간접 노출이 없는가

### Git

- [ ] 이미 추적 중인 파일인지 확인했는가
- [ ] .gitignore가 실제 대상에 적용되는가
- [ ] staged diff를 직접 확인했는가
- [ ] 커밋 메시지에 민감정보가 없는가
- [ ] 이전 이력에 같은 정보가 없는가

### Remote

- [ ] Remote 소유자와 URL이 맞는가
- [ ] Public·Private 범위를 확인했는가
- [ ] 다른 팀원에게 정말 필요한 문서인가
- [ ] 로컬에만 둘 수 있는 자료는 아닌가
- [ ] 전송 후 되돌리기 어렵다는 점을 고려했는가

## 차단됐을 때

외부 전송이 차단되면 먼저 어느 단계까지 성공했는지 구분해야 했다. 파일 수정, staging, commit과 push는 같은 상태가 아니었다.

그다음 민감한 원본뿐 아니라 간접 기록을 확인하고, 안전한 대안을 선택했다. 로컬에 보관하거나, 개인 정보를 제거한 공식 버전을 만들거나, 사용자가 직접 실행해야 하는 명령을 안내할 수 있었다.

가장 중요한 원칙은 우회하지 않는 것이었다.

> 보안 검토는 작업을 방해하는 마지막 문이 아니라, 무엇을 팀의 공식 기억으로 내보낼지 다시 묻는 단계였다.
