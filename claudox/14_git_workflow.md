# 14. Git 워크플로우와 커밋 메시지

처음에는 Git에 남기는 것이 많을수록 안전하다고 생각했다. 코드도 문서도 대화 기록도 모두 저장하면 잃어버릴 일이 없을 것 같았다. 실제로 이 프로젝트는 Markdown 문서가 상품이었고, 결정 과정 자체가 교육 자료였으니 기록은 중요했다.

하지만 작업이 쌓이면서 다른 질문이 생겼다. 기록할 가치가 있다는 것과 GitHub까지 보내야 한다는 것은 같은 의미일까?

## Git과 Remote는 같은 것이 아니었다

Git은 로컬에서도 완전한 버전 관리 도구였다. commit은 로컬 이력을 만들고, push는 그 이력을 외부 Remote에 전달했다. 이 구분을 알고는 있었지만 업무 문서를 다루기 시작하자 의미가 달라졌다.

소스 코드와 함께 README, 설치 가이드, 기술 결정 문서를 공유하는 것은 자연스러웠다. 반면 개인 메모, AI와의 관계 맥락, 작성 중인 기획과 일회성 스크린샷까지 같은 방식으로 올리는 것은 필요하지 않았다.

더 중요한 사실도 있었다. Git은 파일별 원격 보관함이 아니었다. 로컬 전용 문서를 커밋에 포함한 뒤 다른 파일만 계속 push하는 방식은 오래 유지하기 어려웠다. push는 파일 하나가 아니라 커밋과 브랜치의 이력을 전달하기 때문이다.

그래서 문서를 만들 때부터 세 등급으로 나누기로 했다.

| 등급 | 의미 | 처리 |
|---|---|---|
| Official | 다른 환경에서도 재현·이해·검증에 필요한 공식 지식 | Git 추적 후 Remote 공유 |
| Local | 현재 사람과 AI의 작업 기억, 초안과 내부 맥락 | .local에 보관 |
| Ephemeral | 다시 만들 수 있는 일회성 산출물 | 작업 후 정리 |

분류 기준은 파일 확장자가 아니었다. 같은 Markdown이라도 설치 문서는 Official이 될 수 있었고, 개인적인 Day 1 기록은 Local이 될 수 있었다.

## Handoff도 생명주기가 있었다

LEEDOX 통합 홈을 만들 때 기획 요청서, 제품 문서, 전달 가이드, 기존 화면과 결과 보고서가 하나의 handoff 패키지에 모였다. 구현 중에는 모두 필요했다. 그러나 구현이 끝난 뒤에도 전체 패키지를 Remote에 남겨야 하는 것은 아니었다.

최종 코드는 Platform 저장소의 공식 이력이 됐다. 반면 내부 기획, 중간 보고와 전체 화면 원본을 포함한 handoff는 두 저장소의 .local/handoff/completed 아래에 보관했다. 장기적으로 필요한 결정과 교훈만 별도의 사례 문서로 정제했다.

이 구분 덕분에 Remote는 업무 중간 파일의 창고가 되지 않았고, 그렇다고 과정의 증거를 잃지도 않았다.

## Private도 로컬은 아니었다

Private 저장소는 공개 범위가 좁지만 내 컴퓨터 안의 메모장과 같지 않았다. push하는 순간 데이터는 외부 서비스로 전송된다. 권한을 가진 계정이 접근할 수 있고, 이력에 들어간 정보는 나중에 파일을 삭제해도 이전 커밋에 남을 수 있었다.

따라서 API 키, 토큰, 비밀번호와 공개가 승인되지 않은 개인정보는 처음부터 Git에 넣지 않는 것이 원칙이 됐다. 개인적인 기억은 CONTEXT.local.md와 .local 폴더로 분리했다.

## 좋은 커밋은 결과의 경계를 보여준다

문서 분류가 정리되자 커밋 메시지의 역할도 선명해졌다. 메시지는 작업 중 나눈 모든 대화를 설명하는 곳이 아니었다. 공식 결과가 무엇인지, 왜 이 변경이 하나의 단위인지 알려주는 표지였다.

좋은 커밋을 위해 다음을 확인했다.

- 하나의 커밋이 하나의 설명 가능한 목적을 갖는가
- 중간 파일과 개인 메모가 섞이지 않았는가
- 변경된 코드와 필요한 공식 문서가 함께 있는가
- 메시지가 무엇을 바꿨는지 간결하게 말하는가
- push 전에 staged diff와 공개 범위를 확인했는가

Git을 잘 쓴다는 것은 모든 것을 남기는 일이 아니었다. 무엇을 팀의 공식 기억으로 남기고, 무엇을 로컬 작업 기억으로 지킬지 결정하는 일이었다.
