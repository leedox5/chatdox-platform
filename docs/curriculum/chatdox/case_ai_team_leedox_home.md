# 사례: LEEDOX 통합 홈과 가상의 AI 팀

> 기획 에이전트와 구현 에이전트를 분리하고 handoff 계약으로 협업한 첫 번째 완주 사례

- 사례일: 2026-07-13
- 결과: Accepted
- Acceptance Criteria: 17/17 충족

## 1. 사례의 의미

LEEDOX 통합 홈 작업은 하나의 AI에게 기획부터 구현과 자체 검토까지 모두 맡기지 않았다. Curriculum 저장소의 Chatdox가 제품 기획과 완료 조건을 정리하고, Platform 저장소의 별도 구현 에이전트가 실제 코드 조사, 구현, 테스트와 결과 보고를 맡았다.

두 에이전트는 같은 Codex 기반일 수 있지만 대화 기억, 저장소 상태와 책임을 자동으로 공유하지 않는 별도의 팀원으로 운영했다.

이 사례는 AI의 기억 단절을 감추기보다 역할 분리, 문서화된 계약과 교차 검토로 바꾸어 실제 팀과 유사한 협업 효과를 만든 과정이다.

## 2. 출발점

기존 홈은 Chatdox 단일 상품과 SaaS 구축 커리큘럼, 구독 가격과 기술 스택에 집중되어 있었다. 새로운 기획에서는 LEEDOX가 상위 브랜드가 되고, 방문자가 Chatdox와 Claudox의 차이를 이해한 뒤 자신에게 맞는 상품을 선택할 수 있어야 했다.

동시에 기존 Chatdox 콘텐츠, 인증, 결제, 문서와 대시보드 기능은 보존해야 했다. 아직 확정되지 않은 가격과 제공 범위는 새 홈에서 고객 약속처럼 표현하지 않는 것이 중요한 제약이었다.

## 3. 역할 분리

### Product Owner / LEEDOX

- 제품 방향과 우선순위 결정
- 기획안과 구현 결과 승인
- 사업적 미결정 사항의 최종 판단

### Chatdox

- 기존 화면과 기획 의도 분석
- 제품 기획 원본 정리
- 포함·제외 범위 정의
- Acceptance Criteria 작성
- handoff 패키지 구성
- 구현 결과와 스크린샷 검토

### Platform Agent

- 현재 코드와 라우팅 조사
- 기존 구조를 활용한 최소 변경안 선택
- 코드 구현과 자동화 테스트
- 반응형·접근성·브라우저 오류 검증
- 결과 보고서와 스크린샷 제공

## 4. Handoff 계약

Chatdox는 구현 에이전트에게 대화 전체를 전달하지 않았다. 대신 구현에 필요한 맥락을 하나의 패키지로 구성했다.

- request.md: 구현 범위, 제외 범위와 완료 조건
- product.md: 제품과 사이트의 기획 의도
- guide.md: 전달, 구현과 검증 원칙
- landing.png: 변경 전 홈 화면

구현 에이전트는 기획을 임의로 재정의하지 않고, 현재 Platform 코드를 먼저 조사한 뒤 결과와 미결정 사항을 보고하도록 요청받았다.

## 5. 구현자의 독립적인 판단

Platform Agent는 기존 홈을 삭제하지 않는 최소 변경안을 선택했다.

- 루트 홈은 LEEDOX 통합 브랜드 홈으로 변경
- 기존 Chatdox 홈은 별도 경로에서 기존 partial을 재사용
- 기존 Claudox 문서 경로와 권한 체계 유지
- 공통 Header와 Footer만 브랜드 구조에 맞춰 확장
- 기존 인증, 결제와 데이터 구조는 변경하지 않음

이는 기획자의 의도를 따르면서도 실제 코드 구조를 조사한 구현자가 기술적 해법을 독립적으로 선택한 사례다.

## 6. R1 결과

첫 구현에서는 다음이 확인됐다.

- LEEDOX 상위 브랜드와 두 상품의 관계 표현
- 기존 Chatdox 콘텐츠 보존
- 확정되지 않은 가격과 성과 수치 미노출
- 로그인 전·후 Header 유지
- 기존 인증·결제·문서·대시보드 경로 보존
- 데스크톱·태블릿·모바일 반응형 렌더링
- 새 통합 테스트 5건, 49 assertions 통과

구현자는 증거가 부족한 세 항목을 억지로 완료 처리하지 않고 보류로 보고했다.

- LEEDOX 브랜드의 즉시 인지 여부
- 실제 키보드 Tab/Enter 접근성
- DevTools Console의 직접 오류 확인

## 7. Chatdox의 교차 검토

Chatdox는 결과 보고서뿐 아니라 데스크톱과 모바일 전체 화면을 직접 검토했다.

전체 기획과 구조는 성공적으로 반영됐지만 모바일의 주요 제목에서 한글 단어가 음절 단위로 끊기는 문제가 발견됐다.

- ''방법을''이 줄 사이에서 분리됨
- ''구축하기''가 단어 중간에서 분리됨

기능 오류는 아니지만 첫 화면과 상품 카드의 완성도에 영향을 주는 문제였다. Chatdox는 전체 구조를 다시 바꾸지 않고 줄바꿈 수정과 보류 검증만 수행하도록 범위를 제한해 R2를 요청했다.

## 8. R2 보완

Platform Agent는 주요 한글 제목에 단어 단위 줄바꿈을 적용했다. 카피, 섹션 순서, 라우팅과 기능은 변경하지 않았다.

추가로 다음을 실제 브라우저 자동화로 검증했다.

- 데스크톱·모바일 Header 순회
- 모바일 메뉴의 키보드 열기
- Hero와 상품 CTA의 Enter 활성화
- FAQ의 키보드 토글
- Footer 링크 순회
- Runtime exception과 Console error 수집
- 가로 overflow와 반응형 레이아웃

수정 후 기존 Rails 테스트와 Tailwind 빌드도 다시 통과했다.

## 9. 최종 결과

R2 검토 후 17개 Acceptance Criteria 전체를 충족한 것으로 판정했다.

- 기획 의도 반영: 충족
- 기존 콘텐츠와 기능 보존: 충족
- 데스크톱·모바일 반응형: 충족
- 키보드 접근성: 충족
- 서버·브라우저 오류 검증: 충족
- 테스트와 결과 증거: 충족

Product Owner와 Chatdox는 결과를 최종 승인했다.

## 10. 이 사례에서 배운 점

### 역할 분리가 자기확증을 줄였다

기획자가 완료를 선언하지 않고 구현자가 테스트 증거를 보고했으며, 다른 에이전트가 화면을 다시 검토했다.

### Acceptance Criteria가 대화를 계약으로 바꿨다

예쁘게 구현했다는 인상 대신 기존 기능 보존, 미확정 정보, 접근성과 테스트를 포함한 검증 가능한 기준으로 완료 여부를 판단했다.

### 보류 판정은 실패가 아니라 품질 장치였다

구현자가 확인하지 않은 사항을 미확인으로 남겼기 때문에 후속 검증 범위를 작고 정확하게 정의할 수 있었다.

### AI의 기억 단절이 문서화를 촉진했다

두 에이전트가 기억을 공유하지 않기 때문에 의도, 범위, 금지 사항과 결과를 명시적으로 기록해야 했다. 이 문서가 팀의 공용 기억 역할을 했다.

### R2는 전체 재작업이 아니었다

첫 결과를 덮어쓰거나 기획을 다시 시작하지 않고, 발견된 문제와 보류 항목만 후속 범위로 전달했다.

## 11. Git 문서 가이드에 따른 마무리

전체 handoff 패키지는 내부 기획, 중간 결과와 대용량 스크린샷을 포함하므로 로컬 완료 증거로 보관한다.

이 공식 사례 문서에는 장기적으로 필요한 결정, 근거, 결과와 학습만 압축했다. Platform의 실제 코드와 테스트는 Platform 저장소의 공식 이력으로 관리하는 것이 적합하다.

구분은 다음과 같다.

| 산출물 | 관리 방식 |
|---|---|
| Platform 코드와 테스트 | Platform 공식 Git 이력 |
| 전체 handoff 원본 | Curriculum 로컬 완료 보관 |
| 중간·전체 스크린샷 | 로컬 증거 |
| 협업에서 얻은 지식 | 이 사례 문서로 정제 |

## 12. 상품 2 Claudox와의 연결

이 사례는 AI 사용법을 설명하는 데 그치지 않는다. AI가 팀에 들어왔을 때 역할, 기억, 요청, 검토와 책임이 어떻게 달라지는지를 실제 작업으로 보여준다.

특히 다음 질문에 구체적인 답을 제공한다.

- AI를 역할이 있는 팀원으로 어떻게 운영하는가?
- 서로 기억을 공유하지 않는 AI에게 맥락을 어떻게 전달하는가?
- 기획자와 구현자의 권한을 어떻게 나누는가?
- AI의 결과를 누가 어떤 증거로 검토하는가?
- 실패나 보류를 어떻게 다음 Revision으로 연결하는가?

따라서 LEEDOX 통합 홈 handoff는 Claudox가 다루는 ''AI가 들어온 뒤 달라진 팀의 일하는 방식''을 보여주는 첫 번째 완주 사례로 활용할 수 있다.

## 13. 한 문장으로 정리

> Chatdox가 제품의 의도와 완료 기준을 설계하고, Platform Agent가 이를 코드와 테스트로 구현하며, Product Owner와 Chatdox가 결과를 다시 검토함으로써 가상의 AI 팀이 하나의 작업 사이클을 완주했다.
