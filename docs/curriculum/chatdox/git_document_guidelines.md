# Git에서 업무 문서를 관리하는 기준

> Markdown을 비롯한 업무 문서를 언제 Git으로 추적하고, 언제 Remote에 공유하며, 언제 로컬에만 둘 것인지 판단하기 위한 가이드

- 작성일: 2026-07-13
- 적용 대상: LEEDOX의 Curriculum, Platform 및 AI 협업 저장소

## 1. 문제의식

프로젝트를 진행하다 보면 소스 코드뿐 아니라 기획서, 요청서, 결과 보고서, 대화 기록, 개인 메모와 스크린샷도 쌓인다. Markdown은 작성하기 쉽기 때문에 모든 문서를 Git에 넣기 쉽지만, 모든 문서가 프로젝트의 공식 이력이나 Remote 공유 대상일 필요는 없다.

중요한 질문은 파일 형식이 아니라 문서의 역할과 독자다.

> 이 문서는 프로젝트를 다른 사람이나 다른 PC에서 재현·이해·검증하는 데 필요한가, 아니면 지금 나와 AI가 작업하기 위해서만 필요한가?

## 2. Git 추적과 Remote 공유는 다르다

Git은 로컬에서도 사용할 수 있다. 파일을 커밋했다고 해서 자동으로 GitHub에 전송되는 것은 아니며, push해야 Remote에 반영된다.

다만 일반적인 Git에서는 파일별로 ''이 파일은 로컬 커밋만 유지하고 나머지만 계속 push한다''는 방식을 장기적으로 운영하기 어렵다. push는 선택한 파일이 아니라 커밋과 브랜치 이력을 전송하기 때문이다.

한 커밋에 로컬 전용 문서가 포함되어 있으면 그 커밋을 push할 때 문서도 함께 Remote로 간다. 따라서 문서를 만들 때부터 공식 공유 대상과 로컬 전용을 분리하는 것이 안전하다.

## 3. 세 가지 문서 등급

| 등급 | 의미 | Git 추적 | Remote |
|---|---|:---:|:---:|
| Official | 프로젝트의 공식 지식과 증거 | 예 | Push |
| Local | 개인·AI 작업 기억, 초안과 내부 맥락 | 아니요 | Push 안 함 |
| Ephemeral | 일회성 임시 산출물 | 아니요 | 작업 후 정리 |

### Official

다른 환경에서 clone하거나 다른 팀원이 받아도 필요한 문서다.

- 프로젝트 요구사항
- 아키텍처와 기술 의사결정
- 설치, 운영과 배포 가이드
- 코드와 함께 변경되어야 하는 명세
- 팀 공용 작업 규칙
- 재현에 필요한 구현 요청서
- 검증된 handoff 요청과 결과
- 감사, 회고와 교육에 필요한 공식 처리 이력
- 저장소와 함께 잃어서는 안 되는 지식

예시:

- README.md
- docs/
- QA/
- SETUP/
- handoff/archive/
- ARCHITECTURE.md
- CONTRIBUTING.md

### Local

현재 작업을 돕지만 프로젝트의 공식 기록은 아닌 문서다.

- 개인 메모와 사고 정리
- 작성 중인 초안
- AI와의 상세 대화 기록
- 아직 확정되지 않은 기획
- 일회성 작업 지시 패키지
- 중간 스크린샷
- 로컬 환경에만 필요한 정보
- 개인정보와 관계 맥락
- Remote 보존 가치보다 노출 위험이 큰 문서

예시:

- .local/context/
- .local/drafts/
- .local/handoff/
- scratch/
- CONTEXT.local.md
- notes.private.md

### Ephemeral

작업이 끝나면 다시 볼 가치가 낮고 재생성 가능한 파일이다.

- 이미지 변환 중 생성한 파일
- 임시 비교 자료
- 복사 전달을 위한 중복본
- 테스트용 임시 데이터
- 자동 생성 가능한 중간 산출물

## 4. Remote에 포함할 판단 기준

다음 질문 중 여러 항목이 ''예''라면 Official 문서로 관리할 가치가 높다.

- 다른 팀원이나 AI 에이전트가 반드시 읽어야 하는가?
- 다른 PC에서 작업을 이어가는 데 필요한가?
- 코드 변경의 이유와 범위를 설명하는가?
- 설치, 실행 또는 검증을 재현하는 데 필요한가?
- 결정의 책임과 근거를 남겨야 하는가?
- 시간이 지난 뒤에도 검색하고 참조할 가능성이 높은가?
- 이 문서가 없으면 프로젝트를 잘못 이해할 위험이 있는가?

다음에 해당하면 로컬 전용을 우선 고려한다.

- 현재 세션에서만 필요한가?
- 아직 확정되지 않은 생각인가?
- 공식 문서와 내용이 중복되는가?
- 개인정보나 내부 전략이 포함되는가?
- 다른 사람에게는 맥락보다 노이즈가 되는가?
- 결과가 이미 다른 공식 문서에 반영됐는가?
- 잘못 공개됐을 때 얻는 이익보다 위험이 큰가?

경계가 애매하면 먼저 로컬에 둔다. 나중에 공식 문서로 승격하는 것은 쉽지만, Remote Git 이력에 들어간 정보를 완전히 제거하기는 어렵다.

## 5. 업무 처리 문서

서비스 데스크 요청, 작업 기록과 결과 보고서는 다음 기준으로 판단한다.

### Remote에 남길 가치가 높은 경우

- 여러 팀원이 진행 상태를 공유해야 한다.
- 코드 변경과 요청을 연결해야 한다.
- 결정의 책임과 배경을 남겨야 한다.
- 다른 환경에서 이어서 처리해야 한다.
- 교육 콘텐츠나 공개 프로세스의 일부다.
- 구현 결과와 테스트 증거를 보존해야 한다.

### 로컬만으로 충분한 경우

- Product Owner와 특정 AI 세션 사이의 임시 조율이다.
- 작은 문구 수정처럼 일회성 작업이다.
- 개인정보나 내부 사업 전략이 포함된다.
- 결과가 이미 공식 기획서나 docs에 반영되어 있다.
- 티켓이 쌓이면서 중요한 기록을 가린다.
- 요청 패키지가 전달 후 더 이상 필요하지 않다.

모든 변경을 티켓으로 만드는 방식은 프로세스를 배우는 초기에는 유용하지만 장기적으로 기록 비용과 노이즈가 커질 수 있다. 명시적으로 필요한 요청만 발행하는 No Ticket 기본값은 이를 조정하는 한 방법이다.

## 6. Handoff 문서의 생명주기

handoff는 작성 중, 구현 중과 완료 후의 가치가 다르다.

권장 상태:

- handoff/inbox/: Curriculum에서 준비 중인 로컬 패키지
- handoff/active/: Platform이 구현 중인 패키지
- handoff/archive/: 완료 후 공식 보존하기로 결정한 기록

### 공식 사례로 남길 때

요청과 결과에서 장기적으로 필요한 부분만 보존한다.

- request.md
- result.md
- 필요한 최종 스크린샷

중복된 기획 원본과 대용량 중간 스크린샷은 공식 위치를 참조하거나 제외할 수 있다.

### 일회성 전달 수단일 때

패키지는 로컬 전용으로 유지하고, 최종 결정과 구현 결과만 docs, QA 또는 변경 이력에 반영한다.

## 7. Private 저장소의 의미

GitHub Private 저장소는 공개 저장소보다 접근 범위가 좁지만 로컬 전용 저장 공간은 아니다.

- 데이터가 외부 서비스로 전송된다.
- 권한을 가진 계정과 팀원이 접근할 수 있다.
- 서비스의 백업, 로그와 캐시 정책의 영향을 받는다.
- 공개 범위나 권한이 실수로 바뀔 수 있다.
- Git 이력에 들어간 정보는 완전히 제거하기 어렵다.

따라서 Private은 비공개 협업 공간이지 개인 메모장과 같지 않다.

## 8. 민감정보

다음 정보는 Markdown 문서나 Git 이력에 기록하지 않는다.

- API 키와 접근 토큰
- 비밀번호와 세션 정보
- 인증서와 개인키
- 결제수단과 민감한 고객 데이터
- 공개가 승인되지 않은 개인정보

민감정보를 커밋한 뒤 다음 커밋에서 삭제해도 이전 이력에는 남는다. 처음부터 Git에 넣지 않는 것이 원칙이다.

## 9. 피해야 할 방식

### push 직전에 매번 파일 제외

반복적인 수동 제외는 실수하기 쉽다. 한 번 잘못 stage하거나 commit하면 Remote에 포함될 수 있다.

### skip-worktree를 보안 수단으로 사용

skip-worktree는 로컬 설정 파일의 변경 노이즈를 줄이는 기능에 가깝다. 개인정보 보호나 Remote 제외 장치가 아니다.

### 이미 추적 중인 파일을 .gitignore만으로 제외

.gitignore는 추적되지 않은 파일에 적용된다. 이미 추적 중인 파일은 ignore 규칙만 추가해도 추적이 중단되지 않는다.

### 모든 대화를 원문으로 보존

대화 원문은 맥락을 많이 담지만 검색성, 노출 위험과 유지 비용이 크다. 공식 저장소에는 결정, 근거, 결과와 미결정 사항을 요약해 남기는 편이 낫다.

## 10. 권장 저장소 구조

    repo/
    ├─ docs/                 공식 교육 콘텐츠
    ├─ QA/                   확정된 결정과 근거
    ├─ SETUP/                재현 가능한 설정
    ├─ handoff/
    │  ├─ archive/           공식 보존한 인계 기록
    │  └─ inbox/             로컬 전용 작업 패키지
    ├─ service-desk/         명시적으로 접수한 공식 요청
    └─ .local/               개인·AI 작업 기억과 초안
       ├─ context/
       ├─ drafts/
       ├─ screenshots/
       └─ scratch/

프로젝트 성격에 따라 handoff 전체를 로컬로 두고 archive만 별도 공식 문서에 반영할 수도 있다.

## 11. .gitignore 예시

    .local/
    scratch/
    *.local.md
    *.private.md
    handoff/inbox/
    handoff/**/screenshots/

스크린샷이 교육 자료나 완료 증거로 필요하면 예외적으로 추적한다. 단순 화면 확인용이면 로컬에 둔다.

이미 추적 중인 파일에는 이 규칙이 자동으로 적용되지 않는다는 점을 기억한다.

## 12. 문서 승격 절차

로컬 초안을 공식 기록으로 승격할 때 다음을 확인한다.

1. 문서의 장기 독자와 목적을 명시한다.
2. 개인 메모와 불필요한 대화 원문을 제거한다.
3. 결정, 근거, 결과와 미결정 사항을 분리한다.
4. 개인정보와 내부 전략의 공개 범위를 확인한다.
5. 중복된 공식 문서가 있는지 확인한다.
6. 파일 위치와 이름을 공식 구조에 맞춘다.
7. Remote에 전송해도 되는지 최종 검토한다.

## 13. LEEDOX 기본 운영 원칙

1. 문서 형식이 아니라 역할과 독자로 분류한다.
2. 다른 환경의 재현과 협업에 필요한 문서만 Official로 관리한다.
3. 개인·AI 맥락과 확정 전 기획은 Local을 기본값으로 한다.
4. 임시 산출물은 재생성 가능하면 보존하지 않는다.
5. 민감정보는 처음부터 Git에 넣지 않는다.
6. Private 저장소도 외부 전송이라는 점을 기억한다.
7. handoff는 완료 후 장기 보존 가치를 다시 판단한다.
8. 공식 저장소에는 대화 원문보다 결정과 검증 결과를 남긴다.
9. 경계가 애매하면 로컬에 두고 필요할 때 승격한다.
10. push 전에는 커밋에 포함된 문서와 공개 범위를 확인한다.

## 14. 한 문장 기준

> 다른 사람이나 다른 PC가 프로젝트를 재현·이해·검증하는 데 필요하면 Git과 Remote에 남기고, 지금 나와 AI의 작업에만 필요하면 로컬에 두며, 다시 만들 수 있는 일회성 자료라면 정리한다.
