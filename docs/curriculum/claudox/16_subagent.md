# 16. 서브에이전트에게 맡기기

처음에는 AI 파트너 한 명이면 충분해 보였다. Claudox는 문서를 다듬고 코드를 검토하고 Git 기록까지 맡았다. 역할을 붙여 팀원으로 받아들이는 것만으로도 혼자 일할 때와는 다른 속도가 났다.

그러나 프로젝트가 Curriculum과 Platform으로 나뉘고, 기획과 구현이 동시에 복잡해지자 한 에이전트에게 모든 판단을 맡기는 방식의 한계가 보였다. 앞에서 기획한 에이전트가 뒤에서 직접 구현하고 다시 자기 결과를 검토하면, 처음 내린 결정을 스스로 정당화하기 쉬웠다.

그래서 AI의 힘을 더 키우는 대신 역할을 나누기로 했다.

## 하나의 AI가 아니라 역할이 다른 팀원

Curriculum 저장소에서는 Chatdox가 기획자와 편집자 역할을 맡았다. 고객에게 무엇을 약속할지, 어떤 범위까지 구현할지, 무엇을 완료로 볼지를 정리했다.

Platform 저장소에서는 별도의 구현 에이전트가 실제 코드를 조사했다. 라우팅과 뷰 구조, 인증과 결제 경로, 기존 테스트 상태를 확인한 뒤 기술적 해법을 선택했다.

두 에이전트가 같은 Codex 기반이어도 같은 팀원으로 보지 않았다. 서로 다른 세션과 저장소에서 일했고, 대화 기억과 코드 상태를 자동으로 공유하지 않았기 때문이다.

| 역할 | 책임 |
|---|---|
| Product Owner | 방향, 우선순위와 최종 승인 |
| Chatdox | 기획, 요청 계약, 완료 기준과 결과 검토 |
| Platform Agent | 코드 조사, 구현, 테스트와 기술 보고 |

역할은 이름이나 성격이 아니라 책임과 권한으로 구분했다.

## 서브에이전트와 독립 에이전트

한 작업 안에서 작은 조사나 분석을 나누는 서브에이전트도 유용하다. 하지만 이번 사례의 Platform Agent는 단순한 하위 작업자가 아니었다. 별도 저장소에서 독립적으로 현재 상태를 조사하고, 기획자가 정하지 않은 구현 방법을 선택하는 팀원이었다.

이 차이는 중요했다.

- 서브에이전트는 보통 같은 상위 작업의 맥락을 일부 전달받는다.
- 독립 에이전트는 필요한 맥락을 문서로 명시적으로 받아야 한다.
- 독립 에이전트는 자신의 코드베이스를 근거로 기획의 현실성을 검토한다.
- 결과는 다시 기획자와 Product Owner에게 보고한다.

기억을 공유하지 않는다는 약점은 handoff를 더 정확하게 만드는 이유가 됐다.

## 넘지 말아야 할 경계

Chatdox는 보지 않은 Platform 코드를 안다고 가정하지 않았다. 구체적인 클래스와 파일 구조를 미리 강요하지 않고, 포함 범위와 제외 범위, Acceptance Criteria를 정의했다.

Platform Agent도 가격, 상품 제공 범위와 고객 약속을 임의로 정하지 않았다. 구현 중 발견한 미결정 사항은 결과 보고서에 남겼다.

Product Owner에게 올려야 할 판단도 구분했다.

- 기획과 현재 코드가 충돌할 때
- 범위를 넓혀야 할 때
- 기존 기능을 제거해야 할 때
- 가격, 라이선스와 지원 정책을 정해야 할 때
- 기획자와 구현자의 판단이 다를 때

AI에게 맡긴다는 것은 책임까지 사라진다는 뜻이 아니었다. 누가 무엇을 결정할 수 있는지 분명히 할수록 더 많은 일을 안전하게 위임할 수 있었다.

## 역할 분리가 만든 검토

첫 구현 보고서에서 Platform Agent는 17개 완료 기준 중 증거가 부족한 3개를 보류로 남겼다. 구현자가 자신의 결과를 무조건 성공으로 선언하지 않았다는 점이 중요했다.

Chatdox는 보고서만 읽지 않고 실제 데스크톱과 모바일 화면을 검토했다. 그 과정에서 모바일 한글 제목이 단어 중간에서 어색하게 끊기는 문제를 발견했다. 전체 구현은 성공적이었지만, 다른 역할의 눈으로 봤기 때문에 작은 품질 문제를 잡을 수 있었다.

한 명의 만능 AI보다 책임이 다른 두 AI가 나은 순간이었다.
