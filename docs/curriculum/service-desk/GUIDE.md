# Service Desk 사용법

유저, 팀원 등 관련된 자들의 요구사항을 정리하고 그 프로세스를 관리하기 위한 폴더.

## 범위

여기는 **명시적으로 접수된 요청**만 다룬다. `CLAUDE/` 챕터 시스템의 TOC 커버리지 규칙(모든 대화 내용을 챕터에 남긴다)과는 다르다 — 채팅에서 나눈 모든 이야기를 자동으로 여기에 트래킹할 필요는 없다. `new` 명령으로 직접 요청을 만들었을 때만 여기 남는다.

**티켓 발행 기준 (임시, 프로세스 정착 전까지)**: 지금은 문서/코드에 어떤 변경이든 생기면 크기와 상관없이 일단 티켓으로 남긴다 (2026-07-12부터, REQ 0012 계기로 확정). 판단이 애매하면 Claudox가 먼저 Tommy에게 묻고, Tommy가 결정한다. 프로세스가 안정되면 이 기준을 다시 완화할 수 있다.

## 새 요청 만들기

터미널에서 `service-desk/` 안에 있는 스크립트를 실행하면 `requests/_FORM.md`를 복사해 다음 ID로 새 요청을 만들어준다.

- Git Bash: `./new.sh` (또는 `bash new.sh`)
- PowerShell: `./new.ps1`

매번 전체 경로를 안 치고 `new` 한 마디로 실행하고 싶다면, 셸 프로필에 alias/함수를 등록하면 된다.

## 워크플로우: 폴더 이동이 아니라 Status 필드

모든 요청 파일은 `requests/` 폴더 하나에 그대로 머문다. 진행 상태는 파일을 옮기는 게 아니라 헤더의 `Status` 필드 값을 바꾸는 것으로 표현한다.

```
New          새로 접수됨
In Progress  처리 중
Completed    Claudox가 처리를 마침
Confirmed    Leedox가 관여해서 확인함 (직접 재현했든, 검토 후 승인했든)
```

(예전엔 `01_new/` → `02_in_progress/` → `03_completed/`로 파일을 물리적으로 옮겼는데, `Confirmed` 상태를 폴더 이동 없이 도입해보니 굳이 파일을 옮길 필요가 없다는 게 드러나서 REQ 0007로 이 구조 전체를 정리했다.)

## 요청 파일 작성 형식

파일명은 `NNNN.md` (4자리 ID, 예: `0001.md`), `service-desk/requests/` 안에 만든다. 헤더(ID~Status)와 `Description`/`Job` 내용을 각각 `text` 코드펜스로 감싼다. 콜론을 맞추려고 준 공백이 필드마다 달라서(4칸 이상인 줄도, 미만인 줄도 섞여있다), 감싸지 않으면 일부는 의도치 않게 들여쓰기 코드블록으로, 나머지는 한 문단으로 뭉개져 렌더링이 필드마다 제각각이 된다. 코드펜스로 감싸면 이 문제가 통째로 사라진다.

````text
```text
         ID : 0001
       Date : YYYY.MM.DD
  Requester : 요청자 이름
    Subject : 한 줄 제목
     Status : New | In Progress | Completed | Confirmed
 Visibility : Public | Private
```

Description :
```text
요청 내용 상세 (여러 줄 가능)
```

Job :
```text
0001 Claudox YYYY.MM.DD HH:MM
무엇을, 어떻게 처리했는지
```
````

- 새 요청은 `requests/_FORM.md`를 복사해서 다음 ID 번호로 만든다 (직전 최대 ID + 1). `_FORM.md` 자체는 빈 템플릿이니 지우거나 번호를 매기지 않는다.
- 처리를 시작하면 `Status`를 `In Progress`로 바꾼다.
- 완료되면 `Status`를 `Completed`로 바꾸고 `Job`을 채운다.
- `Confirmed`는 **Leedox가 관여했을 때만** 붙인다 — 직접 손으로 재현해봤든(예: REQ 0006), 검토 후 "이것도 Confirmed 처리해줘"라고 명시적으로 승인했든(예: REQ 0001~0004) 상관없다. 핵심은 Claudox 혼자 판단해서 스스로 올리지 않는다는 것 — Claudox가 자체적으로 재확인만 한 것(예: REQ 0007의 첫 시도)은 `Completed`에 머무른다. Leedox의 확인 내용은 새 처리 ID로 `Job`에 이어 적는다.
- 요청은 삭제하지 않는다 — 어떤 상태든 `requests/`에 그대로 남겨 기록으로 유지한다.
- 새 요청이 생기거나 상태가 바뀌면 [dashboard.md](dashboard.md)를 갱신한다 — 손으로 고치는 대신 아래 스크립트를 실행하면 된다.

## dashboard.md 자동 갱신

`requests/*.md`를 스캔해서 `dashboard.md`를 통째로 다시 만들어주는 스크립트.

- Git Bash: `./status.sh` (또는 `bash status.sh`)
- PowerShell: `./status.ps1`

각 요청 파일의 `ID`/`Date`/`Subject`/`Status`를 읽어서 상태별 건수와 표를 다시 계산하므로, 손으로 개수를 세거나 표를 다시 쓸 필요가 없다. 새 요청을 만들거나 `Status`를 바꾼 뒤에는 이 스크립트만 실행하면 된다.

### Requester: 폼을 작성한 사람이 아니라 아이디어를 낸 사람

`Requester`는 실제로 폼을 작성해 접수한 사람이 아니라, **최초로 아이디어를 제공한 사람**이다. 예를 들어 채팅 중에 Claudox가 개선안을 제안하고, Tommy가 "이걸 티켓으로 만들어줘"라고 해서 Claudox가 직접 폼을 작성했더라도, 아이디어를 낸 사람이 Claudox이므로 `Requester : Claudox`다. 반대로 Tommy가 낸 아이디어를 Claudox가 대신 폼으로 작성했다면 `Requester : Leedox`다 — 누가 타이핑했는지가 아니라 누구 생각에서 시작됐는지를 적는다. (2026-07-12부터 적용되는 규칙 — 이전에 접수된 요청은 소급 정정하지 않는다.)

### Job 첫 줄: 처리 ID (요청 ID와는 다른 번호)

`Job` 블록 첫 줄의 4자리 숫자는 **요청 ID가 아니라 "처리 ID"**다. 형식은 `<처리ID> Claudox <완료 일시>`.

- 처리 ID는 **그 요청 안에서** 몇 번째 처리인지를 나타내는, 요청별로 독립된 카운터다. 요청이 처음 처리될 때는 항상 `0001`부터 시작한다.
- 하나의 요청이 여러 번 처리될 수도 있다 (예: 완료 후 검증, 또는 다시 열려서 추가 작업). 그런 경우 같은 요청 파일 안에 `Job` 항목이 늘어나고, 그 요청 안에서만 `0002`, `0003`...으로 증가한다. 요청 ID(파일명)는 바뀌지 않는다.
- **완료 일시는 실제 시각을 확인해서 적는다** (2026-07-12, REQ 0011부터 적용 — 이전 요청의 시각은 소급 정정하지 않는다). Claudox는 실시간 시계에 직접 접근하지 못하므로, 셸에서 `date` 명령을 실행해 나온 값을 그대로 쓴다. 대화 흐름상 그럴듯하게 시간을 지어내지 않는다.

### SUB 요청: 덩어리가 큰 요청을 쪼개기 (REQ 0014부터)

한 요청이 서로 다른 종류의 결정(예: 기술 구현 + 비즈니스 방향 + 컨텐츠 정책)을 한 번에 담고 있어서 추적하기 어렵다면, 별도 ID를 가진 **SUB 요청**들로 쪼갠다.

- SUB 요청도 똑같이 `requests/NNNN.md`로 새 ID를 받는다 — 별도 번호 체계나 접두사는 없다.
- SUB 요청의 `Description` 첫 줄에 `상위 요청: 000X`를 적어 부모를 명시한다.
- 부모(메인) 요청의 `Description`이나 `Job`에 딸린 SUB 요청 ID들을 나열한다.
- **부모 요청은 모든 SUB 요청이 끝나야 완료된다** — SUB가 하나라도 New/In Progress면 부모는 `In Progress`에 머무른다.
- SUB 요청끼리도 서로 순서가 있을 수 있다 (예: 방향 결정 SUB가 끝나야 구현 SUB를 시작할 수 있음) — `Description`에 "선행 요청: 000Y" 형태로 적어둔다.

### Visibility: 이 요청을 웹에 공개할지 (REQ 0016부터)

`service-desk/`가 웹에 퍼블리싱되면서(REQ 0014), 요청마다 공개 여부를 구분할 필요가 생겼다.

- **기본값은 `Public`** — 기술 문제 해결, 프로세스 개선, 문서 작업 같은 요청은 그 자체로 "AI와 어떻게 일하는가"를 보여주는 콘텐츠라 공개한다.
- **`Private`은 예외적으로** 가격/수익/상품 전략처럼 사업 판단이 담긴 요청에만 붙인다 (예: REQ 0014, 0015 — 구독 가격, 상품 구조 결정). 경쟁사에 전략이 그대로 노출되는 걸 막기 위함이다.
- 새 요청을 만들 때 `_FORM.md` 기본값은 `Public`이다. 사업 전략성 내용이면 접수 시점에 `Private`로 바꾼다. 애매하면 Tommy에게 묻는다 (티켓 발행 기준과 동일한 원칙).
- `Private` 요청도 `dashboard.md`에는 그대로 집계되지만(내부용이므로), 웹 퍼블리싱 구현(SUB-C) 쪽에서는 `Private` 요청을 걸러내고 보여줘야 한다.
