# 10. 코드 리뷰를 부탁하다

Platform Agent의 결과 보고에는 테스트 통과와 네 장의 스크린샷이 있었다. 새 가격과 조건이 표시됐고 기존 문서 뷰어도 살아 있었다. 문서만 읽으면 완료라고 판단하기 쉬웠다.

하지만 Tommy는 보고서의 결론을 승인하는 대신 화면을 직접 확인했다.

## 완료 보고를 다시 보다

Claudox 상세페이지의 목차에는 20개 챕터가 모두 “완성”으로 표시돼 있었다. 실제 진행표에서는 11개만 완료였고 나머지는 제목과 미작성 표시만 있는 파일이었다. 구현은 파일이 존재한다는 사실을 내용이 완성됐다는 뜻으로 해석했다.

경로도 어색했다. Chatdox는 `/chatdox`에서 상품을 소개했지만 Claudox는 `/claudox`가 문서 뷰어였고 상품 소개는 `/claudox/intro`에 있었다. 기능은 작동했지만 두 상품의 정보 구조가 달랐다.

Chatdox 상품 구성표에는 요청하지 않은 커뮤니티, 오피스아워와 코드리뷰 크레딧도 들어갔다. 결과 보고는 이를 미결정 사항으로 남겼지만 화면은 이미 제공 예정 기능처럼 약속하고 있었다.

## 서로 다른 종류의 리뷰

| 검토 관점 | 발견한 것 |
|---|---|
| 기획 원본 대조 | 요청하지 않은 운영 지원 기능 |
| 데이터 의미 확인 | 파일 존재와 콘텐츠 완성도의 혼동 |
| 사용자 흐름 확인 | 두 상품의 비대칭 URL 구조 |

코드 리뷰는 문법이나 성능만 보는 일이 아니었다. 요청서, 데이터의 의미와 사용자의 이동 경로가 실제 화면에서 일치하는지 확인하는 일이기도 했다.

## 승인 대신 근거를 남기다

Tommy는 “마음에 들지 않는다”는 말로 돌려보내지 않았다. 잘못된 화면, 기대한 상태, 보존해야 할 동작과 Acceptance Criteria를 세 항목으로 정리했다. R1 전체는 유지하고 필요한 변경만 R2 델타 요청으로 만들었다.

AI가 만든 결과를 다른 AI와 사람이 교차 검토하면서 역할이 분리됐다. 구현자의 자신감이나 보고서의 완성도 대신 기획과 실제 화면이 최종 판단 기준이 됐다.
