# 17. 보안 & OWASP

> "화면에 메뉴가 안 보인다"와 "실제로 접근이 막혀 있다"는 다릅니다. 이 장은 실제로 이 서비스에서 발견됐던 인가(authorization) 누락 사고를 사례로, 눈에 보이는 것과 실제로 안전한 것의 차이를 다룹니다.

---

## 📋 이번 장에서 할 수 있게 되는 것

1. "메뉴에 안 보이니까 안전하다"는 착각을 알아챈다
2. 로그인 여부와 권한 여부를 서버에서 매번 확인하는 구조를 만든다
3. 비밀번호·API 키 같은 민감정보가 로그나 화면에 그대로 새지 않게 한다
4. 보안 문제를 발견했을 때 "고쳐줘"에서 끝내지 않고 "다른 곳에도 같은 문제가 있는지" 같이 확인한다

---

## 1️⃣ 지금 우리에게 필요한 것

기능을 만들 때 보통 "이 화면은 관리자만 보여야 한다"처럼 화면 단위로 생각하기 쉽습니다. 하지만 실제 위험은 화면이 아니라 **주소(URL)와 데이터**에 있습니다. 메뉴에서 링크를 숨겨도, 그 주소를 아는 사람이 직접 입력하면 그냥 열리는 경우가 실제로 있었습니다.

---

## 2️⃣ 실제 사례 — 메뉴만 숨겨져 있던 화면

모바일 메뉴를 손보던 중, 화면 캡처를 다시 살펴보다가 이상한 걸 발견했습니다. 서비스데스크(운영 문의) 화면 중 일부 항목은 "비공개(Private)"로 표시돼 있었는데, 그 안에는 가격 전략처럼 외부에 보이면 안 되는 내용이 들어 있었습니다.

주소를 직접 입력해서 확인해보니, **로그인하지 않은 상태에서도 그 화면이 그대로 열렸습니다.**

코드를 확인해보니 원인은 명확했습니다 — 이 화면을 담당하는 컨트롤러에 애초에 "로그인했는가", "권한이 있는가"를 확인하는 코드 자체가 없었습니다. 다른 관리자 화면들은 전부 공통 규칙을 따르고 있었는데, 이 화면 하나만 그 규칙을 안 따르고 있었던 겁니다.

> 서비스데스크 화면 중 비공개로 표시된 티켓이 로그인 없이도 열리는 걸 발견했어. 이거 당장 막아줘. 그리고 다른 관리자 화면들은 어떻게 권한을 확인하고 있는지 봐서, 똑같은 방식으로 통일해줘.

이 요청에서 중요한 부분은 뒷문장입니다. 발견한 문제 하나만 고치는 게 아니라, **"제대로 하고 있는 곳의 방식과 통일해달라"**고 요청하면 AI는 그 프로젝트 전체에서 같은 실수가 또 있는지까지 같이 확인하게 됩니다.

---

## 3️⃣ AI에게 요청하기 — 예방 차원의 점검

사고가 난 뒤가 아니어도 미리 점검을 요청할 수 있습니다.

> 지금 이 프로젝트에서 로그인 없이 접근하면 안 되는 화면들 중에, 실제로 권한 확인이 빠진 곳이 있는지 전체적으로 훑어봐줘.

> 비밀번호나 API 키 같은 민감한 값이 에러 메시지나 로그에 그대로 남는 곳이 있는지도 확인해줘.

---

## 4️⃣ 확인하기

| 확인할 것 | 방법 | 성공 기준 |
|---|---|---|
| 로그인 없이 관리자/운영 화면 접근 | 로그아웃 상태로 해당 주소 직접 입력 | 로그인 화면으로 이동한다 |
| 일반 회원으로 관리자 전용 화면 접근 | 일반 회원 계정으로 해당 주소 직접 입력 | 접근이 막힌다(관리자 화면이 안 보인다) |
| 비공개(Private) 콘텐츠 노출 여부 | 권한 없는 계정/비로그인 상태로 비공개 항목 목록·상세 확인 | 목록에도 안 뜨고, 주소를 직접 알아도 안 열린다 |
| 민감정보 로그 노출 | 일부러 실패시켜보고 에러 로그 확인(예: 잘못된 값으로 요청) | 비밀번호/키 값이 그대로 로그에 안 남는다 |

---

## 5️⃣ 막히면 이렇게 다시 요청하세요

> "메뉴에는 안 보이는데 걱정돼요" →
> 이 화면이 메뉴에는 안 보이는데, 주소를 직접 알면 들어가지는 거 아니야? 실제로 서버에서 권한을 확인하고 있는지 검증해줘.

> "어떤 화면들을 점검해야 할지 모르겠어요" →
> 로그인이 필요한 화면들이 전부 어디 있는지 목록으로 뽑아줘. 그중에 권한 확인 코드가 없는 곳이 있으면 알려줘.

> "에러 메시지에 이상한 정보가 보여요" →
> 이 에러 메시지에 비밀번호 같은 값이 그대로 보이는 것 같아. 민감한 정보는 가리고 나머지만 남기게 고쳐줘.

---

## ✅ 챕터 17 체크리스트

- [ ] 로그인이 필요한 모든 화면이 실제로 로그인 여부를 서버에서 확인한다
- [ ] 관리자 전용 화면은 관리자 여부까지 서버에서 확인한다
- [ ] "메뉴에 안 보인다"와 "접근이 막혀 있다"를 구분해서 둘 다 확인했다
- [ ] 비밀번호/API 키 같은 값이 로그나 에러 메시지에 그대로 남지 않는다
- [ ] 문제를 하나 발견하면, 비슷한 다른 화면도 같이 점검했다

## 🎯 핵심 원칙

| 원칙 | 설명 |
|------|------|
| 숨기는 것과 막는 것은 다르다 | 링크를 안 보여주는 건 사용자 경험이지 보안이 아니다 — 주소를 직접 알아도 안 되게 서버에서 막아야 진짜 보안이다 |
| 클라이언트를 믿지 않는다 | 화면에서 보낸 값, 화면이 보여주는 상태를 그대로 믿지 않고 서버가 다시 확인한다(9장의 결제 검증과 같은 원칙) |
| 하나 발견하면 전체를 훑는다 | 같은 실수가 다른 곳에도 있을 가능성이 높다 — 발견한 문제 하나만 고치고 끝내지 않는다 |
| 민감정보는 로그에도 남기지 않는다 | 디버깅에 필요한 정보와 유출되면 안 되는 정보를 구분해서, 후자는 화면은 물론 로그에서도 가린다 |

<summary>💡 더 알아보기 — 궁금한 분만 (기술적인 내용, 몰라도 무방합니다)</summary>

이 프로젝트의 관리자 화면들은 공통 컨트롤러(`Admin::BaseController`)를 상속받아, 로그인 여부(`authenticate_user!`)와 권한 여부(`authorize :admin, :access?`, Pundit 기반)를 매번 자동으로 확인합니다. 새 관리자 화면을 만들 때 이 컨트롤러를 상속만 하면 별도로 권한 체크 코드를 반복해서 쓸 필요가 없습니다 — 실제 사고가 났던 화면은 이 공통 규칙을 상속하지 않고 따로 만들어졌던 것이 원인이었습니다.

민감정보 로그 노출을 막는 예시로는, 운영 DB 백업 기능을 만들 때 실패 원인을 로그로 남기되 비밀번호만 골라서 가린 사례가 있습니다.

```ruby
def redact(text, password)
  return text if password.blank?
  text.gsub(password, "[REDACTED]")
end
```

이렇게 하면 "왜 실패했는지" 진단에 필요한 정보(호스트, 에러 문구 등)는 로그에 남기면서, 그 자리에 있을 수 있는 비밀번호 값만 정확히 지울 수 있습니다. 통째로 로그를 안 남기면 안전하긴 해도 원인 진단이 불가능해지고, 통째로 다 남기면 진단은 쉽지만 위험합니다 — 이 둘 사이에서 "민감한 값만 골라서 가리는" 절충이 실전에서 자주 필요합니다.
