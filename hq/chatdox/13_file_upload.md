# 13. 파일 업로드

> 이 장 제목은 원래 "파일 업로드 (Active Storage)"였습니다. 그런데 실제로 이 프로젝트는 Active Storage를 쓰지 않습니다. 왜 안 썼는지, 대신 뭘 했는지를 실제 사례로 다룹니다 — 정답처럼 보이는 방법이 항상 우리 상황에 맞는 건 아니라는 것도 이 장의 내용입니다.

---

## 📋 이번 장에서 할 수 있게 되는 것

1. 문서/콘텐츠에 스크린샷 같은 이미지가 실제로 보이게 만든다
2. "이미지를 어디에, 어떻게 저장할 것인가"를 우리 서비스 상황에 맞게 AI와 함께 판단한다
3. 아무 파일이나 열람되지 않도록 안전하게 서빙한다
4. 배포 환경의 특성(재배포 시 파일이 사라질 수 있다는 것) 때문에 생기는 함정을 미리 안다

---

## 1️⃣ 지금 우리에게 필요한 것

커리큘럼 문서에 실습 스크린샷을 넣고 싶어졌습니다. 마크다운에 이미지 문법(`![설명](경로)`)을 써봤는데, 화면엔 아무것도 뜨지 않았습니다.

원인을 확인해보니 이유가 두 가지였습니다.

- 이미지 파일을 실제로 불러올 수 있는 주소(URL) 자체가 없었다
- 설사 주소가 있어도, 화면을 그리는 과정에서 보안을 위해 `<img>` 태그 자체를 걸러내고 있었다

---

## 2️⃣ AI에게 요청하기

> 문서 본문에 스크린샷을 넣고 싶은데 이미지가 전혀 안 떠. 마크다운에 이미지 문법을 써도 아무것도 안 보여. 확인해서 실제로 이미지가 뜨게 해줘.

AI 파트너가 원인 두 가지(서빙 경로 없음, 보안 필터가 이미지 태그를 지움)를 확인하고 나면, 둘 다 함께 처리해달라고 명확히 못 박습니다.

> 그 두 가지 다 고쳐줘. 그리고 이미지 주소로 아무 파일이나(예를 들어 서버의 설정 파일 같은 것) 열람할 수 있으면 안 되니까, 그 부분도 꼼꼼하게 막아줘.

---

## 3️⃣ 판단 포인트 — 왜 Active Storage를 쓰지 않았나

Rails에서 파일 업로드라고 하면 보통 Active Storage를 먼저 떠올립니다. 하지만 이 프로젝트에서는 다른 선택을 했습니다. 이유를 이해하려면 먼저 상황을 봐야 합니다.

- 이미지를 올리는 사람은 운영자(콘텐츠 작성자)뿐이다 — 일반 사용자가 아무 때나 업로드하는 기능이 아니다
- 이미지는 문서 콘텐츠와 함께 저장소(git)에서 같이 관리되고 있다
- 서비스는 Railway에 배포돼 있는데, **Railway의 기본 컨테이너 디스크는 재배포될 때마다 초기화된다** — 별도로 "Volume"(영구 디스크)을 마운트하지 않는 이상, 오늘 서버에 저장한 파일이 내일 배포 후엔 사라질 수 있다

이런 상황에서 Active Storage(기본 `:local` 서비스)로 이미지를 "업로드"받으면, 그 파일은 컨테이너의 임시 디스크에 저장됩니다. 재배포 한 번이면 흔적도 없이 사라질 수 있다는 뜻입니다. 이걸 막으려면 원래는 S3 같은 외부 저장소를 따로 붙여야 하는데, 이번 요구사항(운영자가 콘텐츠 이미지를 넣는 것) 치고는 과합니다.

> 💡 실제로 이 판단은 나중에 또 한 번 등장합니다 — 운영팀 문의 화면에 "스크린샷 첨부" 기능을 추가해달라는 요청이 왔을 때도 똑같은 질문(Railway 디스크가 휘발성이라는 것)이 걸림돌이 됐고, 그 기능은 저장 방식을 먼저 정하기 전까지 보류했습니다. **"업로드"가 필요해 보여도 매번 Active Storage가 정답은 아니라는 것**, 그리고 배포 환경의 특성이 설계에 직접 영향을 준다는 것을 보여주는 사례입니다.

그래서 이번 경우엔 훨씬 단순한 방법을 택했습니다 — 이미지 파일을 콘텐츠와 함께 git 저장소에 커밋하고, 그걸 그대로 읽어서 보여주는 서빙 경로만 만드는 것.

---

## 4️⃣ 확인하기

| 확인할 것 | 방법 | 성공 기준 |
|---|---|---|
| 정상 이미지 표시 | 문서에 이미지 경로 넣고 화면 확인 | 이미지가 실제로 보인다 |
| 없는 파일 요청 | 존재하지 않는 파일명으로 직접 주소 접속 | 404가 뜬다(다른 에러나 빈 화면 아님) |
| 엉뚱한 파일 접근 시도 | 이미지 폴더 밖의 파일을 가리키는 주소로 접속 시도(`../../` 같은 패턴) | 차단된다(그 파일 내용이 노출되지 않는다) |
| 재배포 후에도 유지 | 배포를 한 번 더 진행한 뒤 같은 이미지 다시 확인 | 여전히 정상적으로 보인다(git으로 관리되므로) |

## 5️⃣ 막히면 이렇게 다시 요청하세요

> "이미지 문법을 썼는데 그냥 글자로 보여요" →
> 마크다운에 이미지 문법을 썼는데 화면엔 이미지가 아니라 글자 그대로 보여. 실제 이미지 태그로 변환되고 있는지 확인해줘.

> "이미지가 깨져 보여요" →
> 이미지가 뜨긴 하는데 깨진 아이콘만 보여. 파일 자체가 손상된 건 아닌지, 서빙 경로가 정확한지 확인해줘.

> "배포했더니 있던 이미지가 사라졌어요" →
> 어제까지 잘 보이던 이미지가 오늘 배포 후에 안 보여. 이미지 파일 저장 방식이 재배포 때 사라지는 구조는 아닌지 확인해줘.

## 🎯 핵심 원칙

| 원칙 | 설명 |
|------|------|
| 유명한 방법이 항상 우리 상황의 정답은 아니다 | Active Storage가 표준이지만, 이번 상황(운영자 전용, 배포 환경의 디스크 휘발성)엔 더 단순한 방법이 맞았다 |
| 배포 환경의 특성을 먼저 안다 | "재배포하면 파일이 사라질 수 있다"는 사실 하나가 설계 전체를 바꾼다 |
| 공개 경로는 항상 의심한다 | 로그인 없이도 열리는 주소라면, 의도한 파일 밖은 절대 못 열리게 별도로 확인해야 한다 |
| 판단을 미루지 않는다 | "업로드 기능 나중에 필요하면 그때 정하지" 하고 미루면, 정작 필요할 때 저장 방식부터 다시 고민해야 한다 |

<summary>💡 더 알아보기 — 궁금한 분만 (기술적인 내용, 몰라도 무방합니다)</summary>

이미지 서빙 라우트는 컨트롤러에서 파일 경로를 조립할 때 `Pathname#expand_path`로 최종 경로를 정규화한 뒤, 그 경로가 반드시 이미지 폴더 하위에 있는지(`start_with?`) 확인합니다. `../../` 같은 경로 조작으로 폴더를 벗어나려는 시도를 이 단계에서 막습니다. 디렉터리 자체를 여는 것도 막기 위해 `File.file?`로 진짜 파일인지 추가로 확인합니다.

```ruby
def serve_chapter_image(images_dir, filename)
  images_dir = images_dir.expand_path
  path = images_dir.join(filename).expand_path

  if path.to_s.start_with?("#{images_dir}/") && File.file?(path)
    send_file path, disposition: "inline"
  else
    head :not_found
  end
end
```

라우트를 만들 때 한 가지 실수하기 쉬운 부분이 있습니다 — 파일명에 점(`.png`)이 들어있으면 Rails가 이를 "포맷 지정자"로 오인해서 확장자를 잘라버릴 수 있습니다. 라우트에 `format: false`를 명시해야 파일명이 그대로 전달됩니다.

화면을 그리는 쪽에서는 `sanitize()` 헬퍼가 안전을 위해 허용하는 태그 목록을 미리 정해두는데, 기본적으로 `img` 태그가 그 목록에 없습니다. 이미지를 보여주려면 `tags`에 `img`를, `attributes`에 `src`/`alt`를 명시적으로 추가해야 합니다 — 이 하나가 빠지면 이미지 태그 자체가 화면에 그려지기 전에 조용히 삭제됩니다.
