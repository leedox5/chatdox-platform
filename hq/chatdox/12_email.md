# 12. Email & 알림

> 회원가입, 비밀번호 재설정처럼 "메일이 실제로 도착해야 완성되는" 기능들이 있습니다. 이 장에서는 로컬에서 메일을 눈으로 확인하는 법과, 실제 서비스에서 진짜 메일이 나가게 만드는 법을 AI 파트너와 함께 다룹니다.
> 여러분이 할 일은 "메일이 왔는지 안 왔는지"를 직접 확인하고, 안 왔다면 그 사실을 그대로 전달하는 것뿐입니다.

---

## 📋 이번 장에서 할 수 있게 되는 것

1. 비밀번호를 잊었을 때 재설정 메일이 실제로 도착하게 만든다
2. 로컬 개발 중에는 진짜 메일 서버 없이도 메일 내용을 눈으로 바로 확인한다
3. 실제 서비스(운영 환경)에서는 진짜 메일 발송 서비스를 연결한다
4. "화면엔 성공했다고 뜨는데 메일이 안 온다"는 조용한 실패를 알아채고 고친다

---

## 1️⃣ 지금 우리에게 필요한 것

회원가입, 로그인 기능(7장)을 만들 때 사용한 Devise에는 "비밀번호를 잊으셨나요?" 기능이 이미 내장돼 있습니다. 화면도 있고, 버튼도 눌립니다. 문제는 **그 뒤가 비어있을 수 있다**는 것입니다 — 실제로 메일을 내보내는 설정을 아무도 안 채워놨다면, 사용자는 "이메일을 확인해 주세요"라는 성공 메시지만 보고, 받은편지함엔 영원히 아무것도 오지 않습니다.

이건 겉으로는 전혀 티가 안 납니다. 에러 화면도 없고, 콘솔에 빨간 글씨도 안 뜨는 경우가 많습니다. 그래서 이번 장은 "기능을 새로 만드는" 장이 아니라 **"이미 있는 기능이 진짜로 끝까지 동작하는지 직접 확인하는"** 장에 가깝습니다.

---

## 2️⃣ AI에게 요청하기

먼저 지금 상태부터 물어봅니다. 바로 고쳐달라고 하기 전에, 뭐가 문제인지부터 같이 확인하는 겁니다.

> 로그인 화면에 "비밀번호를 잊으셨나요?" 링크가 있던데, 이거 실제로 이메일이 가는 상태야? 로컬에서 눌러봤는데 성공 메시지는 뜨는데 메일이 안 왔어. 뭐가 문제인지 확인해줘.

AI 파트너는 코드를 살펴본 뒤 보통 이런 식으로 원인을 짚어줍니다.

- 비밀번호 재설정 라우트, 화면, Devise 기능 자체는 이미 다 켜져 있다
- 문제는 그 뒤: 실제로 메일을 "어떻게 내보낼지"에 대한 설정이 통째로 비어있다
- 로컬은 발송 방법이 지정돼 있지 않아서, 존재하지도 않는 우편함으로 편지를 던지고 있었고 그 실패조차 에러 없이 조용히 삼켜지고 있었다

원인을 확인했으면 이렇게 요청합니다.

> 로컬에서는 실제 메일 서버 없이, 재설정 메일이 왔는지 그 자리에서 바로 확인할 수 있게 해줘. 그리고 실제 운영 환경에서는 진짜로 메일이 발송되도록 설정까지 준비해줘.

---

## 3️⃣ 실제로 겪은 일 — 정작 내가 로그인을 못 했던 밤

이건 실제로 일어났던 일입니다. 개발 중 로컬 서버에 로그인하려는데 비밀번호가 도무지 기억나지 않았습니다. "비밀번호를 잊으셨나요?"를 눌렀고, 화면엔 "이메일을 확인해 주세요"라는 친절한 문구가 떴습니다.

받은편지함엔 아무것도 없었습니다. 스팸함도 비어 있었습니다.

이게 진짜 문제였던 이유는 단순합니다 — **이 상황을 제일 먼저 겪은 사람이 개발자 본인이었다는 것**입니다. 만약 첫 실제 고객이 똑같이 비밀번호를 잊고, 똑같이 오지 않는 메일을 기다리고 있었다면 아무도 몰랐을 수도 있습니다.

이럴 때 할 일은 혼자 코드를 뒤지는 게 아니라, **본 그대로를 전달하는 것**입니다.

> 방금 비밀번호를 잊어서 재설정을 시도했는데, "이메일을 확인해 주세요"라는 성공 메시지는 떴는데 실제로 메일이 안 왔어. 로컬이랑 운영 환경 둘 다 확인해서 진짜로 메일이 가게 고쳐줘.

---

## 4️⃣ 확인하기

코드를 열어볼 필요 없이, 실제로 눌러보고 확인합니다.

| 확인할 것 | 방법 | 성공 기준 |
|---|---|---|
| 로컬에서 재설정 메일 발송 | 로컬 서버에서 "비밀번호를 잊으셨나요?" 시도 | 메일 내용을 그 자리에서 바로 눈으로 확인할 수 있다(별도 메일함 없이) |
| 재설정 링크로 새 비밀번호 설정 | 확인한 메일의 링크 클릭 → 새 비밀번호 입력 | 정상적으로 로그인 화면으로 이동한다 |
| 새 비밀번호로 로그인 | 방금 바꾼 비밀번호로 로그인 | 로그인 성공 |
| 예전 비밀번호로 로그인 시도 | 재설정 전 비밀번호로 로그인 | 실패해야 한다(더 이상 유효하지 않음) |
| 운영 환경 발송 설정 | 배포된 서비스에서 같은 흐름 시도(실제 이메일 계정 필요) | 실제 받은편지함에 메일이 도착한다 |

다섯 가지가 전부 예상대로 되면 이번 장은 끝입니다.

---

## 5️⃣ 막히면 이렇게 다시 요청하세요

> "성공 메시지는 뜨는데 메일이 안 와요" →
> 재설정 요청하면 성공 화면은 뜨는데 실제 메일이 안 와. 발송 자체가 안 되는 건지, 아니면 어디로 가고 있는 건지 확인해줘.

> "메일은 왔는데 링크를 눌러도 이상한 주소로 가요" →
> 메일은 왔는데 링크 주소가 `localhost`나 이상한 도메인으로 돼 있어. 실제 서비스 주소로 가게 고쳐줘.

> "로컬에서는 잘 되는데 운영 환경에서만 안 돼요" →
> 로컬에서는 메일이 잘 보이는데 운영 환경에 배포하니까 안 와. 운영 환경 메일 발송 설정이 제대로 됐는지 확인해줘.

메일 관련 문제는 "발송 자체가 안 되는 것"과 "발송은 됐는데 다른 문제인 것"을 구분하는 게 중요합니다. 어느 쪽인지 모르겠으면 "이런 증상이 있다"고 그대로 전달하는 것만으로 충분합니다 — 원인 구분은 AI가 합니다.

## 🎯 핵심 원칙

| 원칙 | 설명 |
|------|------|
| "성공 메시지"와 "실제로 됐다"는 다르다 | 화면이 성공을 말해도, 최종 결과(메일 도착)를 직접 눈으로 확인하기 전엔 끝난 게 아니다 |
| 조용한 실패가 제일 위험하다 | 에러 없이 그냥 아무 일도 안 일어나는 실패는 아무도 알아채지 못한 채 오래 남을 수 있다 |
| 로컬과 운영은 따로 확인한다 | 로컬에서 되는 것과 실제 서비스에서 되는 것은 별개의 설정이다 |
| 내가 겪은 불편이 제일 좋은 리포트다 | 버그 리포트를 쓸 생각 없이 그냥 쓰다가 겪은 불편이, 실제 고객이 겪을 뻔한 문제를 미리 알려준다 |

<details>
<summary>💡 더 알아보기 — 궁금한 분만 (기술적인 내용, 몰라도 무방합니다)</summary>

Rails의 메일 발송은 `config.action_mailer.delivery_method`라는 설정 하나로 결정됩니다. 이 값이 비어있으면, 메일을 만들긴 하지만 어디로도 보내지 않고 조용히 끝나버립니다.

**로컬 개발 환경**에서는 `letter_opener`라는 gem을 씁니다. 실제 메일 서버 대신, 메일을 만드는 순간 브라우저 새 탭으로 그 메일을 그대로 열어서 보여줍니다.

```ruby
# config/environments/development.rb
config.action_mailer.delivery_method = :letter_opener
config.action_mailer.perform_deliveries = true
```

**운영 환경**에서는 실제 메일 발송 서비스(이 프로젝트는 Resend를 씀, 18장 참고)를 SMTP로 연결합니다.

```ruby
# config/environments/production.rb
config.action_mailer.delivery_method = :smtp
config.action_mailer.smtp_settings = {
  address: "smtp.resend.com",
  port: 587,
  user_name: "resend",
  password: ENV.fetch("RESEND_API_KEY"),
  authentication: :plain,
  enable_starttls_auto: true
}
config.action_mailer.default_url_options = { host: "leedox.up.railway.app" }
```

`default_url_options`도 잊기 쉬운 부분입니다 — 메일 안에 들어가는 "비밀번호 재설정" 링크의 도메인을 결정하는 값인데, 이게 비어있거나 틀리면 메일은 오지만 링크가 엉뚱한 곳(`localhost` 등)을 가리키게 됩니다. 발신자 주소(`mailer_sender`)도 환경변수로 관리해서, 코드에 실제 이메일 주소를 하드코딩하지 않습니다.

</details>
