# 20. 런칭 & 운영

> "다 만들었다"와 "팔아도 된다"는 다른 질문입니다. 이 장은 실제로 이 서비스를 여는 과정 — 이 문서를 쓰고 있는 지금도 진행 중인 과정 — 을 그대로 담습니다. 아직 완전히 끝나지 않은 상태를 굳이 각색하지 않고 보여주는 게, 오히려 이 장의 핵심입니다.

---

## 📋 이번 장에서 할 수 있게 되는 것

1. "기능이 다 됐다"와 "실제로 팔아도 안전하다"를 구분해서 확인한다
2. 런칭 전에 반드시 확인해야 할 항목들을 체크리스트로 만든다
3. 판매를 실제로 켜는 스위치가 여러 겹으로 막혀 있는 이유를 이해한다
4. 문제가 생겼을 때 빠르게 판매를 멈출 수 있는 방법을 미리 준비해둔다

---

## 1️⃣ 지금 우리에게 필요한 것

지금까지 이 커리큘럼을 따라오면서 회원가입, 결제, 대시보드, 보안까지 다 만들었습니다. 그런데 이걸 그대로 켜서 실제 손님을 받아도 될까요?

기능이 동작하는 것과, **실제 돈이 오가는 상황에서 안전한 것**은 다른 질문입니다. 결제가 됐는데 권한이 안 열리면? 같은 결제가 중복으로 처리되면? 서버에 문제가 생겨서 데이터가 날아가면 복구할 방법은 있나?

이런 질문들에 미리 답해두는 게 이번 장입니다.

---

## 2️⃣ AI에게 요청하기 — 런칭 전 종합 점검

한 번에 다 확인해달라고 요청합니다.

> 지금 이 서비스를 실제로 오픈해서 결제를 받아도 되는 상태인지 종합적으로 점검해줘. 결제, 보안, 백업, 약관, 운영 대응까지 다 확인해서, 막고 있는 게 뭐가 있는지 목록으로 정리해줘.

이런 점검은 보통 "막상 열어보니 문제였던" 것보다 "미리 확인해서 다행이었던" 것들을 찾아줍니다. 실제로 이 프로젝트에서도 그랬습니다.

---

## 3️⃣ 실제 사례 — 백업이 하나도 없다는 걸 런칭 직전에 발견

런칭 점검 항목 중 하나로 "데이터베이스 백업이 되고 있는가"를 확인하다가, 지금 쓰고 있는 서버 플랜(Railway Hobby)에서는 **자동 백업 기능 자체가 지원되지 않는다**는 걸 발견했습니다. 즉 지금 상태로 오픈했다가 서버에 문제가 생기면, 결제 기록도 회원 정보도 전부 복구할 방법이 없었다는 뜻입니다.

> 지금 데이터베이스에 백업이 전혀 없다는 걸 발견했어. 정식 백업 기능은 상위 플랜에서만 지원되는데, 일단은 관리자가 수동으로 백업 파일을 받을 수 있는 기능부터 만들어줘.

이 요청 하나로 만든 기능이 실제로 동작하는지 확인하는 과정도 순탄하지 않았습니다 — 백업 파일을 만드는 도구 자체가 서버에 없어서 처음엔 실패했고, 도구를 설치한 뒤에도 서버와 도구의 버전이 안 맞아서 또 실패했습니다. 실패할 때마다 "왜 실패했는지"를 정확히 알 수 있게 로그를 개선해달라고 요청했고, 그 로그 덕분에 정확한 원인(버전 불일치)을 찾아 마지막엔 실제로 백업 파일을 받아서 복원까지 검증했습니다.

> 방금 또 실패했는데, 왜 실패했는지 로그에 안 남아있어. 원인을 알 수 있게 로그를 남기게 해줘. (그 로그로 원인을 확인한 뒤) 이제 원인을 알았으니, 이 문제를 정확히 고쳐줘.

**한 번에 안 되는 게 정상입니다.** 실패할 때마다 "왜 안 됐는지 보이게 해달라"고 요청하고, 보이면 그걸 근거로 다음 요청을 하는 식으로 좁혀가면 됩니다.

## 4️⃣ 런칭 전 확인 체크리스트

| 항목 | 확인할 것 |
|---|---|
| 결제 검증 | 실제 테스트 결제로 성공/실패/취소가 전부 의도대로 동작하는가 |
| 중복 방지 | 같은 결제를 여러 번 시도해도 데이터가 중복 생성되지 않는가 |
| 데이터 백업 | 서버에 문제가 생겼을 때 복구할 방법이 있는가, 실제로 백업 파일을 받아 검증했는가 |
| 약관/정책 | 이용약관, 환불 정책이 실제 운영 방식과 일치하고 실제로 게시돼 있는가 |
| 권한/보안 | 관리자 전용 화면과 비공개 데이터가 실제로 보호되는가(17장) |
| 긴급 중단 방법 | 문제가 생겼을 때 신규 결제만 빠르게 멈출 수 있는 방법이 있는가 |

---

## 5️⃣ 확인하기

| 확인할 것 | 방법 | 성공 기준 |
|---|---|---|
| 판매 스위치가 꺼진 상태 | 판매 활성화 전 상품 페이지 접속 | "준비 중" 문구가 보이고 실제 결제 버튼은 없다 |
| 판매 스위치를 켠 직후 | 관리자 화면에서 판매 여부를 켠다 | 실제 구매 버튼이 나타난다 |
| 첫 실거래 | 스위치를 켠 직후 실제(또는 최소 금액) 결제 1건 진행 | 결제 완료 → 권한(라이선스) 발급까지 정상 동작 |
| 백업 복구 가능 여부 | 받아둔 백업 파일로 실제 복원 시도 | 데이터가 정상적으로 복원된다 |

---

## 6️⃣ 막히면 이렇게 다시 요청하세요

> "런칭해도 되는지 확신이 안 서요" →
> 지금 상태로 정말 오픈해도 되는지 모르겠어. 아직 안 된 것, 확인 안 된 것이 있으면 전부 목록으로 알려줘.

> "문제가 생겼는데 어떻게 멈춰야 할지 모르겠어요" →
> 지금 결제 관련해서 문제가 생긴 것 같아. 신규 결제만 빠르게 멈추고, 이미 진행 중인 결제는 안전하게 처리되게 해줘.

> "백업은 받았는데 이게 진짜 쓸모 있는 파일인지 모르겠어요" →
> 방금 받은 백업 파일이 실제로 복원 가능한 파일인지 확인해줘.

- [ ] 데이터 백업을 실제로 받아서 복원까지 검증했다
- [ ] 약관/정책이 실제로 게시돼 있고 실제 운영 방식과 일치한다
- [ ] 문제가 생겼을 때 신규 판매만 빠르게 멈추는 방법을 알고 있다
- [ ] 판매를 켠 직후 첫 실거래를 직접 지켜봤다(또는 지켜볼 계획이 있다)

## 🎯 핵심 원칙

| 원칙 | 설명 |
|------|------|
| "된다"와 "안전하다"는 다른 질문이다 | 기능이 동작하는 것과, 돈이 오가는 상황에서 실패해도 복구 가능한 것은 별개로 확인해야 한다 |
| 스위치는 겹겹이 둔다 | 판매 여부를 켜는 조건을 하나가 아니라 여러 겹으로 두면, 준비가 덜 된 상태로 실수로 열릴 위험이 줄어든다 |
| 미리 확인해서 다행인 것을 찾는다 | 런칭 전 점검의 목적은 "역시 문제없다"를 확인하는 게 아니라, 열기 전에 다행히 찾아낸 문제를 세는 것이다 |
| 멈추는 방법을 미리 준비한다 | 문제가 생긴 다음에 멈추는 방법을 찾으면 늦다 — 열기 전에 "이럴 땐 이렇게 멈춘다"를 정해둔다 |

이 프로젝트의 판매 스위치는 한 곳이 아니라 여러 겹입니다.

```text
1. 전역 판매 활성화 여부(환경변수)
```

셋 중 하나라도 안 갖춰지면 체크카웃 화면은 실제 결제 버튼 대신 "준비 중" 문구만 보여줍니다. 이렇게 여러 겹으로 만드는 이유는, 배포 실수나 설정 누락 같은 걸로 "의도치 않게 판매가 열리는" 사고를 막기 위해서입니다 — 하나를 깜빡해도 나머지가 막아줍니다.

긴급 중단이 필요할 때는 신규 결제 진입(체크아웃)만 막고, 이미 결제가 진행 중이던 건의 콜백/웹훅 처리는 계속 살려둡니다. 이미 승인된 결제의 후속 처리(라이선스 발급 등)를 막아버리면, 결제는 됐는데 권한이 안 열리는 더 큰 문제가 생기기 때문입니다.

## 마치며

여기까지 오셨다면, 랜딩 페이지 하나였던 프로젝트가 회원가입, 결제, 대시보드, 관리자 화면, 보안 점검까지 갖춘 서비스가 됐습니다. 이 문서를 쓰는 지금도 실제 결제 승인이 완료되지 않아 마지막 스위치는 아직 꺼져 있습니다 — 그것도 그대로 남겨둡니다. 완성은 어느 한순간 "짠" 하고 오는 게 아니라, 이 책의 매 장에서 했던 것처럼 **작게 요청하고, 눈으로 확인하고, 막히면 다시 물어보는** 과정이 쌓여서 만들어진다는 걸 보여주고 싶었기 때문입니다.
