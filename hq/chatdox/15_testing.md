# 15. 테스트

> 원래 이 장 제목은 "테스트 (RSpec)"였습니다. 실제로 이 프로젝트는 처음부터 지금까지 Rails에 기본 내장된 **Minitest**를 씁니다. RSpec을 따로 설치할 필요가 없다는 뜻입니다.
> "지금 되는 것"과 "앞으로도 계속 되는 것"은 다릅니다. 이 장은 그 차이를 실제 사례로 다룹니다.

---

## 📋 이번 장에서 할 수 있게 되는 것

1. 새 기능을 요청할 때 "테스트도 같이"라는 말 한마디의 의미를 이해한다
2. 코드를 고칠 때마다 전체 테스트를 돌려서 예상치 못한 곳이 깨지지 않았는지 확인하는 습관을 들인다
3. 자동 테스트가 놓칠 수 있는 것과, 사람이 직접 눌러봐야만 알 수 있는 것을 구분한다
4. 테스트가 "지금 이대로도 통과하는 코드"를 지키기만 하는 게 아니라, 설계의 위험 신호를 미리 알려준다는 걸 실제 사례로 본다

---

## 1️⃣ 지금 우리에게 필요한 것

기능 하나를 새로 만들거나 고칠 때마다 매번 화면을 눌러서 사람이 직접 확인하는 건 오래 걸리고, 확인을 빼먹기도 쉽습니다. 테스트는 "이 조건에서는 이런 결과가 나와야 한다"는 걸 코드로 적어두고, 몇 초 안에 전부 다시 확인해주는 장치입니다.

이 프로젝트는 Rails가 기본으로 제공하는 **Minitest**를 씁니다. `bin/rails test` 한 줄이면 전체 테스트가 돌아갑니다. 커리큘럼 제목엔 RSpec이라고 적혀 있었지만, 실제로는 Minitest만으로 충분했습니다 — 별도 gem을 더 설치하고 문법을 새로 배울 이유가 없었기 때문입니다.

---

## 2️⃣ AI에게 요청하기

새 기능을 요청할 때 테스트를 같이 요청하는 게 핵심입니다.

> 방금 만든 [기능]에 대한 테스트도 같이 작성해줘. 정상적으로 되는 경우뿐 아니라, 권한이 없는 사람이 시도하는 경우, 존재하지 않는 데이터를 요청하는 경우도 같이 확인해줘.

기존 코드를 고칠 때는 이렇게 요청합니다.

> 이거 고치고 나서 전체 테스트가 통과하는지 꼭 확인해줘. 혹시 기존 테스트 중에 깨지는 게 있으면, 그냥 넘어가지 말고 왜 깨졌는지 원인을 알려줘.

---

## 3️⃣ 실제 사례 — 통과하던 테스트가 알려준 것

### 사례 1: 있어야 할 테스트가 없었던 경우

문서 페이지 하단에 "최종 업데이트" 날짜가 표시되는데, 어느 날 이게 영어로("July 16, 2026") 뜨는 걸 발견했습니다. 원인은 금방 찾았지만, 더 중요한 질문은 "왜 지금까지 아무도 몰랐지?"였습니다.

확인해보니, 비슷한 다른 화면(Claudox 소개 페이지)에는 "날짜가 한글로 뜨는지" 확인하는 테스트가 있었는데, 정작 문제가 생긴 화면(Chatdox 문서 페이지)에는 그 짝이 되는 테스트가 아예 없었습니다. 테스트가 있었다면 이 문제는 배포되기 전에 이미 실패로 잡혔을 겁니다.

> 이 문서 페이지가 영어로 날짜를 보여주는 문제를 고쳐줘. 그리고 비슷한 다른 화면엔 이런 문제를 잡는 테스트가 있는지 확인해서, 없으면 여기에도 만들어줘.

고치는 것과 별개로, **"왜 안 걸렸는지"**까지 같이 물어보는 게 핵심입니다. 같은 문제가 다른 곳에서 또 생기지 않게 막는 방법이기 때문입니다.

### 사례 2: 테스트가 설계 문제를 미리 알려준 경우

체크아웃을 여러 번 시도해도 중복 주문이 안 쌓이게 막아달라는 요청을 처리하던 중, 그 로직을 주문 생성 함수 안에 직접 넣으려던 시도가 있었습니다. 그런데 그렇게 바꾸자마자 전혀 관련 없어 보이던 기존 테스트 6개가 갑자기 실패했습니다.

원인을 보니, 그 기존 테스트들은 "같은 조건으로 여러 개의 독립된 주문을 일부러 만드는" 시나리오(예: 결제 확인 순서가 뒤바뀌어도 문제없는지 확인하는 테스트)였습니다. 중복 방지 로직을 그 함수 안에 넣으면, 이 테스트들이 전제하던 동작 자체가 깨져버리는 것이었습니다.

이건 "테스트를 억지로 맞추자"가 아니라 "이 함수는 원래 여러 곳에서 다른 목적으로 쓰이고 있었다"는 신호였습니다. 그래서 중복 방지 로직은 그 함수 안이 아니라, 실제 체크아웃 화면에서만 쓰이는 별도의 얇은 함수로 옮겼습니다. 결과적으로 기존 테스트는 한 줄도 안 고치고 그대로 통과했습니다.

> 방금 그 수정으로 기존 테스트 6개가 깨졌어. 억지로 테스트를 고쳐서 통과시키지 말고, 왜 깨졌는지부터 설명해줘.

테스트가 실패하면 "테스트를 바꿔서 통과시키는" 방향과 "코드 설계를 다시 생각하는" 방향, 두 가지가 있습니다. 어느 쪽이 맞는지는 실패 이유를 들어봐야 압니다.

## 4️⃣ 확인하기

| 확인할 것 | 방법 | 성공 기준 |
|---|---|---|
| 전체 테스트 실행 | `bin/rails test` | 전부 통과(빨간 글씨 없음) |
| 새 기능에 테스트가 있는가 | 방금 추가한 기능 관련 테스트 파일 확인 | 정상 케이스뿐 아니라 실패/권한 없음 케이스도 있다 |
| 기존 테스트가 이유 없이 깨졌는가 | 수정 전후 테스트 결과 비교 | 관련 없어 보이는 테스트가 깨졌다면, 그 이유를 먼저 이해했다 |

## 5️⃣ 막히면 이렇게 다시 요청하세요

> "테스트를 실행하는 법을 모르겠어요" →
> 지금까지 작성된 테스트를 한 번에 다 실행하는 명령어가 뭐야? 실행해서 결과도 알려줘.

> "테스트가 실패하는데 이유를 모르겠어요" →
> 테스트 몇 개가 실패했는데 에러 메시지를 봐도 이해가 안 돼. 뭐가 문제인지, 코드 문제인지 테스트 자체가 잘못된 건지 확인해줘.

> "테스트는 통과하는데 실제 화면에선 안 돼요" →
> 테스트는 전부 통과하는데 실제로 화면에서 써보면 안 돼. 테스트가 실제 상황을 제대로 확인하고 있는 게 맞는지 다시 봐줘.

<details>
<summary>💡 더 알아보기 — 궁금한 분만 (기술적인 내용, 몰라도 무방합니다)</summary>

Minitest 테스트는 `test/` 폴더 아래에 있고, 기본 형태는 이렇습니다.

```ruby
# test/integration/example_test.rb
require "test_helper"

class ExampleTest < ActionDispatch::IntegrationTest
  test "guest cannot access admin dashboard" do
    get admin_dashboard_path
    assert_redirected_to new_user_session_path
  end
end
```

`test "설명" do ... end` 블록 하나가 테스트 하나입니다. `assert_*` 계열 메서드(`assert_response`, `assert_redirected_to`, `assert_match` 등)로 "이래야 한다"는 조건을 표현합니다. RSpec의 `describe`/`it`/`expect`와 하는 일은 같지만 문법이 더 단순합니다.

전체 테스트는 `bin/rails test`, 특정 파일만은 `bin/rails test test/integration/example_test.rb`로 실행합니다. CI(`.github/workflows/ci.yml`)에서는 현재 정적 분석(Brakeman, bundler-audit, Rubocop)만 자동으로 돌고 있고, `bin/rails test` 자체는 아직 CI에 연결돼 있지 않습니다 — 지금까지는 사람이 로컬/개발 환경에서 직접 실행해서 확인하는 방식에 의존하고 있습니다. 이건 이 프로젝트가 안고 있는 실제 개선 과제 중 하나입니다.

</details>
