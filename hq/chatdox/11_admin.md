# 11. 관리자 대시보드

> 코드를 몰라도 괜찮습니다. 이 장에서는 "회원이 몇 명이고, 누가 결제했고, 누구에게 어떤 권한을 줄지"를 한눈에 보는 화면을 AI 파트너와 함께 만듭니다.
> 여러분이 할 일은 원하는 것을 명확하게 설명하고, 결과를 눈으로 확인하고, 부족하면 다시 요청하는 것뿐입니다.

---

## 📋 이번 장에서 할 수 있게 되는 것

1. 서비스 운영자만 볼 수 있는 화면을 AI에게 요청해서 만든다
2. 전체 회원 수, 관리자 수, 유료 이용자 수를 한 화면에서 확인한다
3. 회원 목록에서 각 사람의 권한을 바꿀 수 있게 한다
4. AI가 처음엔 놓쳤던 위험한 상황(내 권한을 실수로 없애버리는 것)을 스스로 알아채고 막아달라고 요청한다
5. 결과가 기대와 다를 때, 원인을 직접 찾지 않고 AI에게 다시 물어보는 법을 익힌다

---

## 1️⃣ 지금 우리에게 필요한 것

서비스를 운영하다 보면 개발자가 아니어도 궁금해지는 것들이 있습니다.

- 지금 회원이 몇 명이지?
- 그중에 실제로 돈을 낸 사람은 몇 명이지?
- 이 사람은 왜 관리자 권한이 필요하다고 하지? 줘도 되나?

지금까지는 이걸 확인하려면 개발자에게 물어봐야 했습니다. 이번 장에서는 **운영자만 들어갈 수 있는 화면**을 만들어서, 이 질문들에 직접 답할 수 있게 합니다. 일반 회원은 절대 이 화면에 들어올 수 없어야 한다는 것도 함께 요청합니다.

---

## 2️⃣ AI에게 요청하기

파트너에게 말하듯 있는 그대로 요청합니다. 화면 이름, 봐야 할 정보, 지켜야 할 조건을 순서대로 이야기하면 됩니다.

> 관리자만 볼 수 있는 화면을 하나 만들어줘. 전체 회원이 몇 명인지, 그 중에 관리자는 몇 명인지, 지금 유료로 이용 중인 사람은 몇 명인지 숫자로 한눈에 보고 싶어. 최근에 가입한 사람 목록이랑 최근 결제 내역도 같이 보여줬으면 좋겠어. 그리고 이건 중요한데 — 일반 회원은 이 화면에 절대 들어올 수 없어야 해. 로그인 안 한 사람도 마찬가지고.

이렇게 요청하면 AI 파트너는 보통 이런 식으로 결과를 돌려줍니다.

- 새 화면 주소(예: `/admin/dashboard`)를 만들었고, 로그인 화면과 연결했다
- 화면 위쪽에 전체 회원·관리자·유료 이용자·최근 가입자 숫자가 카드 형태로 보인다
- 아래쪽에 최근 가입한 사람과 최근 결제 목록이 나열된다
- 로그인하지 않았거나 관리자가 아니면 이 화면 대신 다른 곳으로 돌려보낸다

여기서 회원 목록으로 한 단계 더 들어갑니다.

> 이 대시보드에서 "사용자관리" 화면으로 들어가서, 각 회원의 권한(일반 회원/관리자)을 직접 바꿀 수 있게 해줘. 목록에는 이름, 이메일, 가입일도 같이 보여주고.

---

## 3️⃣ AI가 놓친 걸 우리가 먼저 알아챘을 때

화면이 만들어졌다고 끝이 아닙니다. 실제로 눌러보다가 이런 생각이 들 수 있습니다.

> "어... 내가 실수로 내 관리자 권한을 일반 회원으로 바꿔버리면 어떻게 되지? 그럼 나 자신도 이 화면에 다시 못 들어오는 거 아니야?"

이런 위험한 상황은 미리 다 예상하고 시작할 필요가 없습니다. 쓰다가 걱정되는 순간 바로 요청하면 됩니다.

> 방금 궁금해졌는데, 관리자가 실수로 자기 자신의 권한을 일반 회원으로 바꿔버리면 어떻게 돼? 그러면 아무도 관리자 화면에 못 들어오는 상황이 될 수도 있을 것 같은데, 그런 실수는 아예 못 하게 막아줘.

이 한 마디로 AI는 "본인의 권한은 자기 자신이 낮출 수 없다"는 안전장치를 추가합니다. 실제 서비스에서 정말 흔하게 일어나는 사고 하나를 대화 한 번으로 막은 셈입니다. 처음부터 완벽한 요청을 하려고 애쓰기보다, **써보면서 걱정되는 걸 그때그때 이야기하는 것**이 더 자연스럽고 더 안전한 결과를 만듭니다.

---

## 4️⃣ 확인하기

코드를 열어볼 필요 없이, 화면으로 직접 확인합니다.

| 확인할 것 | 방법 | 성공 기준 |
|---|---|---|
| 로그인 안 한 상태로 관리자 화면 접근 | 주소창에 `/admin/dashboard` 직접 입력 | 로그인 화면으로 넘어간다 |
| 일반 회원 계정으로 접근 | 일반 회원으로 로그인 후 같은 주소 접속 | 관리자 화면이 아닌 다른 곳으로 넘어간다 |
| 관리자 계정으로 접근 | 관리자로 로그인 | 회원 수·관리자 수·유료 이용자 수가 숫자로 보인다 |
| 회원 권한 변경 | 사용자관리 화면에서 다른 회원의 권한을 바꿔보기 | 목록에 바뀐 권한이 바로 반영된다 |
| 본인 권한 변경 시도 | 내 계정의 권한을 일반 회원으로 바꿔보기 | 변경되지 않고 경고 메시지가 뜬다 |

다섯 가지가 전부 예상대로 되면 이번 장은 끝입니다.

---

## 5️⃣ 막히면 이렇게 다시 요청하세요

에러 메시지를 해석하려고 애쓰지 마세요. 본 그대로, 느낀 그대로 전달하면 AI가 원인을 찾습니다.

> "숫자가 다 0으로 나와요" →
> 대시보드 숫자가 전부 0으로 나와. 실제로는 결제한 사람이 있는데 왜 0인지 확인해서 고쳐줘.

> "일반 회원인데 URL을 직접 치니까 들어가져요" →
> 일반 회원 계정으로 관리자 화면 주소를 직접 입력했더니 그냥 들어가져. 로그인 여부만 확인하는 게 아니라 관리자인지도 서버에서 확실히 확인하도록 다시 만들어줘.

> "권한을 바꿨는데 화면이 그대로예요" →
> 회원 권한을 바꿨는데 목록 화면에서 안 바뀌어 보여. 저장은 잘 됐는지, 화면을 새로고침하면 반영되는지 확인해줘.

화면을 보고 느낀 것을 그대로 전달하는 것 자체가 훌륭한 요청입니다. "왜 이런 에러가 났지"보다 "나는 이런 걸 기대했는데 실제로는 이랬다"가 AI에게는 더 명확한 정보입니다.

---

## ✅ 챕터 11 체크리스트

- [ ] 로그인하지 않은 사람은 관리자 화면에 들어갈 수 없다
- [ ] 일반 회원은 관리자 화면에 들어갈 수 없다
- [ ] 관리자는 전체 회원·관리자·유료 이용자 숫자를 한 화면에서 본다
- [ ] 최근 가입자와 최근 결제 내역이 목록으로 보인다
- [ ] 사용자관리 화면에서 회원 권한을 바꿀 수 있다
- [ ] 관리자가 자기 자신의 권한은 스스로 낮출 수 없다

## 🎯 핵심 원칙

| 원칙 | 설명 |
|------|------|
| 구체적으로 요청할수록 정확한 결과가 온다 | "관리자 화면 만들어줘"보다 "누가, 무엇을, 왜 못 하게"까지 이야기하기 |
| 걱정되면 그 자리에서 바로 물어본다 | 완벽한 요청을 미리 준비하지 않아도, 쓰면서 떠오른 걱정을 바로 전달하면 된다 |
| 화면을 숨기는 것과 막는 것은 다르다 | 메뉴에 안 보여도 주소를 알면 들어갈 수 있다 — "진짜로 못 들어가는지"까지 확인 |
| 되돌리기 어려운 실수는 미리 막아달라고 요청한다 | 본인 권한 해제처럼 한 번 잘못되면 복구가 번거로운 일은 AI에게 안전장치를 명시적으로 요청 |
| 느낀 그대로 전달한다 | 에러 메시지를 해석하려 하지 말고, 기대한 것과 실제 본 것을 그대로 이야기하기 |

이 화면은 `role`이라는 값으로 일반 회원과 관리자를 구분합니다. 관리자 전용 화면들은 "로그인했는가"와 "관리자인가"를 매번 서버에서 함께 확인하는 공통 규칙(`Admin::BaseController`, `AdminPolicy`)을 하나 만들어서 전부 그 규칙을 따르게 합니다. 화면이 늘어나도 이 규칙 하나만 지키면 되기 때문입니다.

회원 권한을 바꾸는 저장 로직에는 "바꾸려는 사람이 본인이고, 그 결과로 관리자 권한이 없어지는 경우"를 미리 걸러내는 조건이 들어갑니다. 사용자가 화면에서 이상한 값을 보내더라도, 서버가 실제로 존재하는 권한 값인지 다시 한번 확인합니다 — 브라우저가 보낸 값을 그대로 믿지 않는다는 뜻입니다.
