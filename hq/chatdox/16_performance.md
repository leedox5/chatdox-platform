# 16. 성능 최적화 & 캐싱

> 회원이 10명일 때 잘 되던 화면이 10,000명이 되면 느려지는 경우가 있습니다. 대부분은 코드 로직이 아니라 **데이터베이스를 몇 번 왔다 갔다 하는지**의 문제입니다. 이 장에서는 가장 흔한 성능 문제와 기본적인 캐싱 전략을 다룹니다.

---

## 📋 목표

1. N+1 쿼리 문제를 알아채고 고친다
2. 자주 조회하는 컬럼에 인덱스를 건다
3. 반복 계산되는 값을 캐싱한다
4. 무거운 작업을 백그라운드로 넘긴다

---

## 1️⃣ N+1 쿼리 — 가장 흔한 성능 문제

목록 화면에서 각 항목마다 연관된 데이터를 또 조회하면, 항목이 10개면 쿼리가 11번(목록 1번 + 각 항목마다 1번씩) 나갑니다. 항목이 1,000개면 1,001번이 됩니다.

```erb
<!-- ❌ N+1 발생 -->
<% Product.all.each do |product| %>
  <%= product.product_offers.count %>개 오퍼
<% end %>
```

```ruby
# ✅ 미리 한 번에 가져오기 (eager loading)
Product.includes(:product_offers).each do |product|
  product.product_offers.size   # 추가 쿼리 없음
end
```

`includes`는 연관된 데이터를 미리 한 번에 조회해서 준비해둡니다. 실제로 이 프로젝트의 관리자 상품 관리 화면(`Admin::Commerce::ProductsController`)도 상품 목록과 각 상품의 오퍼 목록을 같이 보여줘야 해서, `Product.includes(:product_offers).order(:code)`처럼 미리 불러오는 방식을 씁니다.

**확인 방법**: 개발 환경 로그(`log/development.log`)에 같은 형태의 쿼리가 반복해서 찍히는지 보거나, `bullet` 같은 gem으로 N+1을 자동 감지할 수 있습니다.

## 2️⃣ 인덱스 — 자주 찾는 컬럼엔 표시를

```ruby
# 인덱스가 없으면: 테이블 전체를 처음부터 끝까지 훑는다
User.find_by(email: "user@example.com")

# 마이그레이션에서 인덱스 추가
add_index :users, :email, unique: true
```

`WHERE`, `ORDER BY`에 자주 쓰이는 컬럼, 그리고 외래키(`user_id` 등)에는 인덱스를 겁니다. 데이터가 적을 땐 차이가 안 느껴지지만, 테이블이 커지면 인덱스 없는 컬럼으로 조회하는 순간 눈에 띄게 느려집니다. `unique: true` 옵션은 인덱스이면서 동시에 "중복 안 됨"도 DB 레벨에서 보장합니다.

---

## 3️⃣ 캐싱 — 같은 계산을 반복하지 않기

### Fragment 캐싱 (화면 일부 캐싱)

```erb
<% cache @product do %>
  <%= render @product %>
<% end %>
```

`@product`가 안 바뀌었으면 저장해둔 HTML을 그대로 재사용하고, `updated_at`이 바뀌면 자동으로 새로 그립니다(Rails의 캐시 키가 모델의 `updated_at`을 포함하기 때문입니다).

### 값 캐싱

```ruby
def self.total_active_users
  Rails.cache.fetch("stats/total_active_users", expires_in: 10.minutes) do
    User.where(active: true).count
  end
end
```

매번 계산하기 무거운 값(통계, 집계)은 일정 시간 동안 저장해뒀다가 재사용합니다. `expires_in`으로 "이 정도는 오래된 값이어도 괜찮다"는 기준을 정합니다 — 실시간으로 정확해야 하는 값(예: 결제 금액)엔 쓰면 안 됩니다.

---

## 4️⃣ 무거운 작업은 백그라운드로

이메일 발송, 대용량 파일 처리처럼 오래 걸리는 작업을 사용자 요청 안에서 그대로 처리하면, 사용자는 그 작업이 끝날 때까지 화면이 멈춘 것처럼 기다려야 합니다.

```ruby
# ❌ 요청 처리 중 바로 실행 — 느림
UserMailer.welcome_email(user).deliver_now

# ✅ 백그라운드로 넘기고 바로 응답
UserMailer.welcome_email(user).deliver_later
```

`deliver_later`는 실제 발송을 별도의 백그라운드 작업(Active Job)으로 넘기고, 사용자에게는 바로 응답합니다. 무거운 작업일수록 "사용자를 기다리게 하지 않고, 뒤에서 처리한다"는 원칙이 중요합니다.

---

## 5️⃣ 페이지네이션 — 한 번에 다 보여주지 않기

```ruby
# ❌ 전체를 한 번에
@orders = Order.all

# ✅ 페이지 단위로
@orders = Order.order(created_at: :desc).limit(20).offset(params[:page].to_i * 20)
```

목록이 수천 건이 될 수 있는 화면(주문 목록, 회원 목록)은 처음부터 페이지네이션을 고려합니다. `kaminari`, `pagy` 같은 gem을 쓰면 이 계산을 직접 안 짜도 됩니다.

---

## ✅ 챕터 16 체크리스트

- [ ] 목록 화면에서 연관 데이터를 조회할 때 `includes`로 N+1을 막았다
- [ ] 자주 조회/정렬하는 컬럼과 외래키에 인덱스가 있다
- [ ] 자주 반복되는 무거운 계산은 캐싱했다
- [ ] 이메일 발송 등 오래 걸리는 작업은 백그라운드로 넘겼다
- [ ] 대량으로 늘어날 수 있는 목록엔 페이지네이션을 적용했다

---

## 🎯 핵심 원칙

| 원칙 | 설명 |
|------|------|
| 추측하지 말고 측정한다 | 로그나 프로파일링 도구로 실제 느린 지점을 확인한 뒤 최적화한다 |
| N+1이 가장 흔한 범인이다 | 화면이 느려졌다면 제일 먼저 반복문 안의 쿼리부터 의심한다 |
| 캐시는 "오래돼도 괜찮은 값"에만 | 결제 금액처럼 항상 정확해야 하는 값을 캐싱하면 안 된다 |
| 사용자를 기다리게 하지 않는다 | 오래 걸리는 작업은 사용자 요청과 분리해서 뒤에서 처리한다 |
