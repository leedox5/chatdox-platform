# 1. 채독스 전체 구조 이해

> 이 챕터에서는 채독스가 어떤 서비스인지, 어떤 기술로 이루어져 있는지,
> 그리고 앞으로 20개 챕터를 통해 무엇을 만들게 되는지 전체 그림을 이해합니다.

---

## 📌 채독스란?

채독스는 **Chat-GPT + Leedox**의 합성어로, AI Assistant를 활용한 SaaS 서비스입니다.

그런데 이 커리큘럼이 다른 Rails 튜토리얼과 다른 점이 하나 있습니다. **여기 적힌 20개 챕터는 강의를 위해 지어낸 예제가 아니라, 실제로 지금 운영 중인 프로덕션 서비스를 만들며 쌓은 진짜 기록**이라는 것입니다.

- 이 서비스는 지금 이 순간에도 실제로 결제를 받고 라이선스를 발급하고 있습니다.
- 여기 나오는 버그들은 지어낸 실습 문제가 아니라 실제로 터졌던 사고입니다 — 중복 주문이 쌓이던 버그, DB 백업 도구가 4라운드 만에야 정상 동작한 사연, 이미지 파일 딱 1바이트가 깨져서 스크린샷이 통째로 안 보이던 사고, 결제 메일이 막힌 포트 때문에 안 나가던 문제까지. 각 챕터의 "실전 사례" 섹션에서 원인과 해결 과정을 그대로 볼 수 있습니다.
- 처음엔 상품이 하나(Chatdox)였지만, 지금은 같은 결제 인프라 위에서 두 번째 상품(Claudox)까지 함께 판매되고 있습니다. "나중에 상품이 늘어나면 어떻게 하지"가 아니라 실제로 늘어났을 때 코드를 어떻게 바꿨는지를 9장에서 다룹니다.

이 프로젝트를 직접 구현하면서 다음을 경험하게 됩니다:

- Ruby on Rails로 **실전 SaaS 서비스** 구축
- 사용자 인증, 결제, 권한 관리 등 **비즈니스 핵심 기능** 구현
- 배포부터 모니터링까지 **프로덕션 운영** 경험
- 완벽한 설계도가 아니라, **실제로 겪은 버그를 실시간으로 고쳐나가는 과정**

> 💡 이 1장은 로그인 없이 누구나 읽을 수 있는 무료 공개 챕터입니다. 전체 20장과 실제 소스코드, 그리고 AI와 이 서비스를 함께 만들어온 협업 기록(Claudox 시리즈)까지 이어서 보고 싶다면, 상품 페이지에서 가격을 확인하세요.

## 👥 사용자 유형별 화면 및 메뉴 구조

사용자 상태에 따라 처음 진입하는 화면과 노출되는 메뉴가 달라집니다. 메뉴를 숨기는 것만으로 권한을 보호하지 않고, 실제 접근 권한은 8장 `Authorization` 정책이 서버에서 검사합니다.

| 사용자 | 기본 화면 | 주요 메뉴 | 목적 |
|---|---|---|---|
| 게스트 (비로그인) | 랜딩페이지 | 회원가입, 문서(1~2장 공개) | 서비스 소개와 공개 문서 체험 |
| 회원 | 대시보드 | 문서, 마이페이지, 로그아웃 | 가입 후 더 넓은 범위(1~5장)를 둘러보는 단계 |
| 라이선스 보유자 | 대시보드 | 문서, 이용현황, 결제관리, 로그아웃 | 전체 콘텐츠(1~20장) 이용 |
| 어드민 | 관리자 대시보드 | 종합현황, 문서, 사용자관리, 마이페이지, 로그아웃 | 사용자·문서·결제 운영 |

### 사용자별 화면 흐름

```text
게스트         → 랜딩페이지 → 회원가입
회원           → 대시보드   → 일부 문서(1~5장) → 구매하기
라이선스 보유자 → 대시보드   → 전체 문서(1~20장) → 이용현황/결제관리
어드민         → 관리자 대시보드 → 종합현황/문서/사용자관리
```

정확한 접근 등급과 기간은 상품 정책이라 이 문서에서 숫자로 고정하지 않습니다. 세부 규칙과 구현은 다음 문서에서 다룹니다.

- [07장 인증](/docs/7): 로그인·로그아웃과 인증 상태
- [08장 권한](/docs/8): 사용자 등급별 접근 정책
- [09장 결제](/docs/9): 라이선스 결제와 다상품 확장
- [10장 사용자 대시보드](/docs/10): 이용현황과 라이선스 확인

## 🏗️ 전체 아키텍처

```
[사용자 브라우저]
      ↓  HTTPS
[Railway]  ← 배포 플랫폼
      ↓
[Rails 8.1 애플리케이션]
  ├─ 랜딩페이지 / 상품 페이지 (Tailwind CSS)
  ├─ 인증 (Devise, 한국어 지원)
  ├─ 결제 & 라이선스 (PortOne V2 + 수동 계좌이체)
  ├─ 문서 시스템 (Markdown Renderer, 다상품 지원)
  └─ 관리자 대시보드
      ↓
[PostgreSQL]  ← 프로덕션 데이터베이스
      ↓
[외부 서비스]
  ├─ PortOne V2       (카드/간편결제)
  ├─ Active Storage    (파일 저장)
  └─ Resend            (이메일 발송, HTTPS API)
```

## 📁 프로젝트 폴더 구조

Rails 프로젝트를 생성하면 다음과 같은 구조가 만들어집니다:

```
chatdox/
├── app/
│   ├── controllers/        # 요청 처리 (비즈니스 로직)
│   ├── models/             # 데이터베이스 모델
│   ├── views/              # HTML 템플릿 (ERB)
│   ├── helpers/             # 뷰 헬퍼 메서드
│   └── assets/              # CSS, JS, 이미지
├── config/
│   ├── routes.rb           # URL 라우팅 설정
│   └── database.yml        # 데이터베이스 설정
├── db/
│   ├── migrate/             # 데이터베이스 변경 이력
│   └── schema.rb            # 현재 DB 구조
├── Gemfile                  # 라이브러리 목록
└── README.md
```

> 💡 **처음에는 모든 폴더를 이해할 필요가 없습니다.**
> 챕터를 진행하면서 각 폴더의 역할을 자연스럽게 익히게 됩니다.

## 🔄 요청 흐름 이해 (MVC 패턴)

Rails는 **MVC(Model-View-Controller)** 패턴을 따릅니다.

```
사용자가 URL 접속
      ↓
1. Router     → 어떤 Controller로 보낼지 결정
2. Controller → 데이터 처리, Model 호출
3. Model      → 데이터베이스에서 데이터 가져오기
4. View       → HTML 생성 (Controller에서 받은 데이터 사용)
      ↓
사용자에게 HTML 응답
```

**예시:**
```
https://chatdox.com/docs/1 접속
  → Router: docs#show 로 연결
  → Controller: 문서 ID=1 데이터 조회
  → Model: DB에서 문서 가져오기
  → View: 문서 HTML 렌더링
  → 사용자 화면에 표시
```

## 📊 데이터 모델 (핵심 구조)

채독스를 운영하기 위한 핵심 데이터 구조입니다. 실제 서비스는 상품이 하나가 아니라 여럿(Chatdox, Claudox)이라, `Product`를 중심에 두고 설계돼 있습니다 — 자세한 스키마와 이유는 9장에서 다룹니다.

```
User (사용자)
  └─ has_many License, Order

Product (상품: chatdox, claudox)
  └─ has_many ProductOffer   # 기간별 가격 옵션

Order (구매 시도 → 결제 확정 → 라이선스 발급)
  └─ has_many OrderItem, has_one PaymentTransaction

License (실제 접근 권한)
  ├─ starts_on / access_ends_at
  └─ status   # scheduled / active / canceled
```
