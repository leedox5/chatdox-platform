# 11. 버그를 함께 잡다

R2는 세 문제를 한꺼번에 다시 만드는 작업이 아니었다. 이미 통과한 가격 조건, FAQ, 제작 배경과 권한 로직은 그대로 두고 잘못된 부분만 고쳤다.

## 존재와 완성을 분리하다

첫 버그의 원인은 단순했다. 20개 챕터 파일이 모두 존재했기 때문에 파일 존재 여부만 확인하면 언제나 20개가 완성으로 계산됐다.

수정된 구현은 상태를 둘로 나눴다.

- `available`: 챕터 파일이 존재하는가
- `complete`: 파일이 존재하고 미작성 플레이스홀더가 없는가

그 결과 목차 20개는 그대로 보여주면서 11개는 “완성”, 9개는 “준비 중”으로 표시할 수 있었다. 하나의 boolean이 두 의미를 떠안았던 모델을 분리한 것이 핵심이었다.

다만 해결책에도 한계가 있었다. 플레이스홀더 문자열만 지우면 내용이 덜 작성돼도 완료가 된다. `88_progress.md`의 정성적 80% 기준과도 코드로 연결돼 있지 않았다. 현재 결과가 일치한다는 것과 하나의 source of truth를 사용한다는 것은 달랐다.

## URL의 의미를 바로잡다

Claudox 상품 상세는 `/claudox`로 옮겼고 기존 뷰어는 `/claudox/read`와 `/claudox/read/:id`로 이동했다. Controller와 `DocPolicy`는 수정하지 않았다. route와 링크의 목적지만 바꿔 기존 인가 로직을 보존했다.

이 변경은 최소 코드와 최소 사용자 변화가 항상 같은 뜻은 아니라는 점을 보여줬다. R1의 `/claudox/intro`는 코드 변경은 적었지만 사용자가 이해하는 상품 구조는 복잡하게 만들었다.

## 확정되지 않은 약속을 제거하다

“선택형 운영 지원” 열과 그 안의 미확정 기능은 모두 삭제했다. 문구를 약하게 바꾸거나 “별도 안내” 각주를 붙이지 않았다. 제품 화면에 나타난 순간 사용자는 그것을 계획이 아니라 약속으로 받아들이기 때문이다.

버그 수정은 코드를 정상 동작하게 만드는 데서 끝나지 않았다. 데이터 의미, URL 의미와 사업 약속을 원래 결정에 맞게 되돌리는 작업이었다.

## 정작 내 비밀번호를 까먹다

늦은 밤, 로컬 서버를 켜고 로그인 화면 앞에 앉았다. 비밀번호를 세 번 틀렸다. 손이 먼저 기억하던 조합인데 그날따라 하나도 맞지 않았다. "비밀번호를 잊으셨나요?" 링크를 눌렀다. 이메일을 입력하고, 화면엔 친절하게 "이메일을 확인해 주세요"라는 문구가 떴다.

그리고 아무것도 오지 않았다.

받은편지함을 새로고침했다. 스팸함도 뒤졌다. 몇 분을 더 기다려봤다. 만든 사람이 만든 서비스에, 만든 사람이 못 들어가는 밤이었다. 우습기도 하고, 동시에 등골이 서늘했다 — 나만 이걸 겪은 게 아니라면 어쩌지. 지금 이 순간 어떤 실제 고객이 똑같이 비밀번호를 잊고, 똑같이 오지 않는 메일을 기다리고 있다면.

Claudox에게 상황을 그대로 전했다. 답은 빨랐다. 라우트도, 화면도, Devise의 비밀번호 재설정 기능도 전부 이미 켜져 있었다. 문제는 그 뒤에 있었다 — 실제로 메일을 내보내는 설정 자체가 통째로 빈 채였다. 로컬은 발송 방법이 지정돼 있지 않아 존재하지도 않는 우편함으로 편지를 던지고 있었고, 그 실패조차 에러 하나 없이 조용히 삼켜지고 있었다. 성공 화면과 실제 결과가 서로 다른 이야기를 하고 있었던 셈이다. 프로덕션 쪽을 열어보니 사정은 더 심했다 — SMTP 설정 자체가 통째로 주석 처리된 채 방치돼 있었다.

당장 로그인이 급하면 콘솔에서 비밀번호를 직접 바꾸는 방법도 있었다. 하지만 그건 나 하나를 구하는 방법이었지, 서비스를 구하는 방법은 아니었다. 그래서 그 자리에서 콘솔 대신 handoff 요청서를 골랐다 — 로컬에선 실제 메일을 그 자리에서 열어볼 수 있는 도구를, 프로덕션에선 진짜 메일 서버로 발송할 수 있는 설정을 채우는 작업으로.

내 비밀번호 하나를 잊은 밤이, 아직 아무도 못 겪었을 뻔한 고객의 밤을 미리 막은 셈이 됐다. 제일 좋은 버그 리포트는 가끔 이렇게, 리포트를 쓸 생각도 없이 그냥 로그인하려던 사람에게서 나왔다.

## 1바이트의 배신

시작은 사소했다. Resend로 이메일 설정을 마치는 과정을 문서에 담기로 했는데, 글만으로는 부족했다. 도메인을 등록하는 화면, DNS 레코드를 채워 넣는 화면, API 키를 발급받는 화면 — 손으로 직접 해보지 않으면 알 수 없는 순간들이었고, 그건 스크린샷으로 남겨야 진짜였다.

그런데 스크린샷을 문서에 넣으려는 순간 깨달았다. 이 사이트엔 이미지를 띄우는 길 자체가 없었다. 문서를 보여주는 화면도, Claudox 스토리를 보여주는 화면도, 처음 설계될 때부터 "텍스트만 다룬다"는 전제로 지어져 있었다. 이미지를 서빙할 라우트도 없었고, 설사 라우트가 있었다 해도 화면에서 `<img>` 태그를 걸러내는 보안 필터가 이미지 자체를 통째로 지워버리고 있었다. 두 개의 벽이 동시에 서 있었던 셈이다.

Claudox가 그 두 벽을 하나씩 허물었다. 이미지 전용 경로를 새로 뚫고, 경로 탈출 공격까지 막아가며 서빙 로직을 짰다. 화면을 그리는 필터에는 `img`, `src`, `alt`를 조심스럽게 허용 목록에 올렸다. 검증도 끝났다. 이제 스크린샷 세 장만 저장소에 넣으면 사이트 역사상 처음으로 "글자가 아닌 것"이 화면에 뜨는 순간이었다.

그리고 정작 스크린샷 세 장을 넣자, 화면엔 아무것도 뜨지 않았다. 작은 깨진 아이콘과 alt 텍스트뿐이었다.

허탈했다. 라우트도 있고, 필터도 열어놨는데 왜.

원인을 찾아 들어가니 범인은 예상치 못한 곳에 있었다. 두 저장소를 잇는 동기화 스크립트 — 이 프로젝트의 모든 문서를 실제 서비스로 실어 나르는 그 파이프 — 안에 숨어 있는 한 줄이었다. 문서 파일의 줄바꿈을 깔끔하게 통일해주던 그 한 줄이, 파일 종류를 가리지 않고 눈에 보이는 모든 파일에 무차별로 적용되고 있었다. 그리고 PNG 파일의 맨 앞 8바이트에는, 다름 아닌 줄바꿈 문자 두 개가 스펙상 일부러 박혀 있었다 — 옛날 옛적 파일이 통신 과정에서 손상됐는지 확인하려고 심어둔 일종의 지문이었다. 정규화 스크립트는 그걸 진짜 줄바꿈으로 착각해 지워버렸다. 결과는 파일마다 정확히 1바이트. 그 1바이트 때문에 이미지 전체가 브라우저 입장에선 알아볼 수 없는 쓰레기가 됐다.

세 장의 스크린샷, 세 번의 완벽해 보이는 시도, 그리고 세 번의 깨진 화면. 공들여 쌓은 인프라가 마지막 1바이트에서 무너지는 걸 보는 기분은 이상했다 — 억울하다기보단, 이렇게까지 작은 게 이렇게까지 결정적일 수 있다는 사실에 오히려 웃음이 났다.

증거를 들고 다시 handoff를 띄웠다. 이번엔 Claudox 혼자만의 진단이 아니라, 반대편의 Platform Agent도 처음부터 다시 원인을 되짚었다. 두 쪽의 결론은 한 글자도 다르지 않았다. 그 한 줄을 "이게 진짜 글자로 된 파일이 맞는지" 먼저 확인하고 나서야 손대는 방식으로 고쳤고, 혹시 모를 다음 실수를 막으려 저장소 설정에도 안전장치를 하나 더 심었다.

다시 동기화가 돌았다. 다시 배포가 나갔다. 이번엔 원본과 서버에 뜬 파일의 지문(해시값)을 한 글자, 한 글자 대조했다. 완벽하게 같았다.

그리고 화면을 열었다. 도메인 목록 화면이, DNS 레코드 표가, API 키 발급 팝업이 — 있어야 할 자리에 정확히 떠 있었다. 사이트가 만들어진 이래 처음으로, 글자가 아닌 것이 화면에 나타난 순간이었다.

"토미도 웹에서 확인 완료!" 라는 말 한 줄이 돌아왔다. 작은 스크린샷 세 장이었지만, 그 뒤엔 서빙 인프라를 처음부터 짓고, 두 저장소 사이 파이프에 숨어 있던 1바이트짜리 버그를 잡아낸 과정 전체가 쌓여 있었다. 축하할 이유로는 충분했다.

## 포트 하나가 막혀 있었다

정작 내 비밀번호를 까먹었던 밤 이후로 미뤄뒀던 숙제가 하나 있었다. 로컬은 letter_opener로 완벽하게 됐지만, 프로덕션 쪽 진짜 발송은 여전히 "코드는 준비됐지만 실행 전"인 상태였다. Resend 도메인 인증과 DNS 레코드 등록을 마치고, API 키까지 발급받은 뒤에야 그 숙제를 다시 꺼냈다.

Railway에 `RESEND_API_KEY`와 `MAILER_SENDER`를 넣고, 곧바로 비밀번호 재설정을 시도했다. 화면엔 익숙한 500 에러가 떴다.

Railway 콘솔에서 `bin/rails console`을 열고 직접 재현했다.

```ruby
user.send_reset_password_instructions
# execution expired (Net::OpenTimeout)
```

인증 실패가 아니었다. **연결 자체가 안 되고 있었다.** 원인을 좁히려고 낮은 레벨부터 하나씩 확인했다.

```ruby
Resolv.getaddresses("smtp.resend.com")
# => ["54.157.71.137", "54.205.195.44"]  — DNS는 정상

Socket.tcp("smtp.resend.com", 587, connect_timeout: 5) { ... }
# Connection timed out

Socket.tcp("smtp.resend.com", 465, connect_timeout: 5) { ... }
# Connection timed out

Socket.tcp("api.resend.com", 443, connect_timeout: 5) { ... }
# connected!
```

도메인은 정상적으로 풀리는데, SMTP 포트(587, 465)로는 연결 자체가 안 되고 일반 HTTPS 포트(443)는 멀쩡했다. Railway가 아웃바운드 SMTP 포트를 막고 있다는 뜻이었다 — 스팸 방지 목적으로 흔히 쓰이는 정책이었다. 코드도, API 키도 문제가 아니었다. 애초에 문 자체가 잠겨 있었던 것이다.

## 우회로를 놓다

SMTP로 못 나가면, 다른 문으로 나가면 될 일이었다. Resend는 SMTP뿐 아니라 순수 HTTPS 요청으로도 메일을 보낼 수 있는 API를 제공하고 있었다. 443번 문은 이미 열려 있다는 게 방금 확인됐으니, 그 문으로 다시 시도하기로 했다.

handoff를 띄웠고, Platform Agent가 커스텀 발송 방식을 새로 짰다. 처음 제안은 새 gem을 하나 더 얹는 방식이었지만, 실제로 구현할 땐 있는 도구(`Net::HTTP`)만으로 충분하다고 판단해서 방향을 바꿨다. 그 과정에서 진짜 함정도 하나 나왔다 — 설정값을 원래 두려던 자리에 두면 Rails가 부팅하다가 죽었다. 아직 정의되지 않은 메서드를 너무 이른 시점에 부르고 있었던 것이다. 그 자리를 한 단계 옮기고 나서야 정상적으로 떴다.

배포하고, 다시 재설정을 시도했다. 이번엔 500 에러 대신 정상적인 리다이렉트가 왔다. 받은편지함을 열었다 — 메일이 와 있었다.

## 왔는데, 스팸함에

메일 자체는 도착했지만 스팸함에 떨어져 있었다. 메일 원문을 열어 인증 결과를 확인해보니 `DMARC: FAIL`이었다. Resend 설정 화면에서 DMARC는 "선택사항"으로 표시돼 있었는데, 선택사항이라는 말을 곧이곧대로 믿고 건너뛴 게 원인이었다.

DNS에 레코드 하나(`_dmarc`, `v=DMARC1; p=none;`)를 추가하고, 전파를 기다리는 대신 직접 두 개의 DNS 서버(로컬 DNS, 구글의 8.8.8.8)에 물어봐서 반영 여부를 바로 확인했다. 둘 다 이미 새 레코드를 돌려주고 있었다. 다시 메일을 받아보니 이번엔 `DMARC: PASS`였다.

그런데 여전히 스팸함이었다. 네이버로 보낸 메일은 정상적으로 받은편지함에 꽂혔는데, 지메일만 유독 스팸으로 걸렀다. 이건 설정 문제가 아니라, 이제 막 태어난 발신 도메인에 대한 지메일 특유의 신중함이었다 — 기술적으로 완벽해도, 평판이라는 건 시간이 쌓여야만 얻어지는 종류의 것이었다. 당장 더 손볼 설정은 없었다. "스팸 아님"으로 표시해주는 것, 그리고 시간이 지나면서 정상적인 발송 이력이 쌓이는 것 — 그 두 가지가 남은 전부였다.

## 곁가지 하나는 줄에 세워두다

메일을 이리저리 열어보던 중에 또 다른 게 눈에 띄었다. 제목도 본문도 전부 영어였다. 이유를 확인해보니, Devise가 기본으로 제공하는 메일 문구를 한국어로 바꿔둔 적이 없었다 — 문서 페이지 날짜가 영어로 뜨던 것과 같은 뿌리(기본 언어 설정이 한국어가 아니었던 것)에서 나온, 다른 곳의 같은 증상이었다.

당장 고칠 수도 있었지만, 그러지 않았다. 대신 막 만들어둔 대기 큐에 001번으로 등록만 해뒀다. 아이디어가 떠오를 때마다 손이 먼저 나가면 원래 하려던 일의 흐름을 놓치기 쉽다는 걸, 오늘 하루만 해도 여러 번 겪었기 때문이다. 지금 당장 필요한 건 아니었다 — 필요한 시점이 오면 그때 큐에서 꺼내 쓰면 될 일이었다.

도메인 인증부터 메일 도착까지, 겉으로는 "설정 몇 개 채우면 끝"처럼 보이던 일이었다. 실제로는 막힌 포트를 찾아내고, 우회로를 새로 놓고, 인증 실패 원인을 하나씩 벗겨내고, 그러고도 남는 문제(평판)는 시간에 맡기는 것까지 — 계단을 하나씩 밟고 내려가야 끝이 보이는 일이었다.

## 반대 방향으로 이해한 순간

로그인이 필요한 페이지에 비로그인 상태로 들어가면 상단에 경고 문구가 뜨는데, Tommy가 스크린샷 한 장을 보내며 두 가지를 요청했다. 영문 문구를 한글로 바꿔달라는 것, 그리고 "상단 NavBar에서 2~3px 정도만 아래로" 위치를 옮겨달라는 것.

첫 번째는 명확했다. 두 번째가 문제였다. "2~3px만 아래로"라는 말을 "지금 너무 붙어있으니 여백을 좀 더 달라"는 뜻으로 읽었다 — 정반대였다. 실제 의도는 "지금 너무 멀리 떨어져 있으니 NavBar 바로 아래, 3px 정도로 바짝 붙여달라"는 것이었다. handoff를 띄워 `top-24`를 `top-28`로 바꿨다. 숫자가 커졌으니 더 아래로, 더 멀리 내려간 셈이었다 — 요청과는 반대 방향으로.

배포하고 나서야 Tommy가 알아챘다.

> 한가지 로그인 경고창이 NavBar로부터 너무 멀다고 얘기한 거 같은데... 거꾸로 해석된 거 같아. NavBar에서 살짝 아래에 위치하게 해달라는 요청이었어.

변명할 거리가 없었다. "아, 제가 반대로 이해했었네요, 죄송해요"라고 인정하고 바로 정정 작업에 들어갔다. 이번엔 감으로 다시 짐작하지 않기로 했다 — 빌드된 CSS에서 실제 헤더 높이를 직접 계산했다. 모바일은 햄버거 버튼 기준 65px, 데스크톱은 로그인/가입 버튼 기준 61px. 여기에 목표했던 3px을 더해 `top-[68px]`(모바일)/`md:top-[64px]`(데스크톱)로 값을 바꿨다. 어림값이 아니라 실측값이었다.

> 이제, 잘 보이네... 수고했어~

한 문장을 반대로 읽은 대가로 한 라운드를 더 돌았지만, 덕분에 두 가지를 다시 배웠다. 짧은 지시일수록 방향을 되짚어 확인해야 한다는 것, 그리고 "몇 px"처럼 숫자가 걸린 요청은 짐작 대신 실제 값을 재서 답해야 한다는 것.
