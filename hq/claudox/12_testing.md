# 12. 테스트 코드를 짜다

R2가 끝난 뒤 통합 테스트는 7개 시나리오와 83개 assertion을 통과했다. 하지만 숫자만으로 모든 품질을 증명했다고 말하지 않았다.

## 요구사항을 assertion으로 바꾸다

새 테스트는 R1에서 실제로 틀렸던 지점을 고정했다.

- `/chatdox`에 단품 조건이 보이는가
- 미확정 운영 지원 문구가 사라졌는가
- `/claudox`가 `완성 11 / 20`을 표시하는가
- CH01은 완성이고 CH06은 준비 중인가
- `/claudox/read`와 개별 챕터 경로가 응답하는가
- LEEDOX 메인 홈은 가격을 계속 숨기는가

테스트는 페이지가 200을 반환하는지 보는 수준을 넘어, 이전 검토에서 발견된 오류가 다시 들어오지 않도록 요구사항을 실행 가능한 문장으로 바꿨다.

## 변경하지 않았다는 것과 검증했다는 것

Platform은 `ClaudoxController`와 `DocPolicy`를 수정하지 않았다. 이는 기존 권한이 보존됐을 가능성을 높이는 좋은 정적 근거였다.

그러나 자동 테스트가 실행한 것은 게스트가 읽을 수 있는 01번 챕터였다. 트라이얼, 구독자와 관리자가 3~20번 챕터를 각 권한 수준에 맞게 읽는지는 실행 검증하지 않았다. “정책 코드를 건드리지 않았다”와 “모든 역할의 권한을 확인했다”는 같은 문장이 아니었다.

## 화면 캡처와 브라우저 검증

1440px 데스크톱과 390px 모바일 전체 화면 PNG가 결과 증거로 남았다. Headless Chrome으로 화면을 생성했고 서버 로그에서 새 오류가 없음을 확인했다.

반면 실제 Tab 순서, 키보드만으로 모든 CTA를 사용하는 과정, Safari와 Firefox, 실제 모바일 기기는 확인하지 않았다. 코드에 `focus-visible` 클래스가 있다는 사실도 실제 키보드 검증을 대신하지 못했다.

## 테스트 보고의 원칙

1. 어떤 명령을 실행했는지 적었다.
2. 어떤 요구사항을 검증했는지 연결했다.
3. 정적 코드 확인과 실행 테스트를 구분했다.
4. 수동 화면 확인과 자동 회귀 테스트를 구분했다.
5. 확인하지 않은 범위를 현재 한계로 남겼다.

테스트의 신뢰는 많은 것을 검증했다고 말할 때보다 어디까지 검증했는지 정확히 경계를 그을 때 더 높아졌다.

## 통과한 테스트가 놓친 버그

한참 뒤, 그 원칙을 정면으로 확인하는 사례가 하나 더 생겼다. GitHub 저장소 접근 기능을 훨씬 단순한 구조로 줄이는 작업이었다.

자동 테스트는 전부 통과했다. `bin/rails test` 결과는 초록색이었고, 이 시나리오 — 이미 계정을 연결한 사용자가 다시 등록을 시도하는 경우 — 도 분명히 테스트 목록에 있었다. 그런데 실제 화면을 흉내 내어 curl로 직접 요청을 보내보니 500 에러가 떨어졌다. `has_one` 연관과 "이미 있으면 막는다"는 제약이 만나는 지점에서, 있는 레코드를 "교체"하려는 시도가 조용히 실패하고 있었다.

같은 시나리오를 검증한다고 적힌 자동 테스트는 왜 이 예외를 못 잡았을까. 원인은 끝내 밝혀지지 않았다. 자동 통합 테스트가 요청을 만드는 방식과, curl로 실제 화면 조작을 흉내 내는 방식 사이에 미묘한 차이가 있었으리라는 추측만 남았다.

여기서 확인된 건 12장 앞부분에서 이미 나온 이야기의 더 날카로운 버전이었다. "테스트가 이 시나리오를 커버한다"는 것과 "이 시나리오에서 실제로 안전하다"는 것은 같은 문장이 아니었다. 자동 테스트가 초록불을 켰다는 사실 하나만으로 안심하지 않고, 실제 화면 조작을 흉내 낸 수동 검증을 병행하는 습관이 있었기에 이 버그는 배포 전에 잡혔다. 원인을 완전히 설명하지 못했어도, "존재 여부를 먼저 확인하고 새로 생성한다"는 방식으로 그 자리를 우회해 고쳤다 — 원인을 다 알아야만 고칠 수 있는 건 아니었다.
