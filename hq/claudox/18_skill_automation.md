# 18. 스킬(Skill)로 반복 작업 줄이기

같은 절차를 몇 번이고 반복하고 있다는 걸 문득 알아챘다. DEV 쪽에 작업을 넘길 때마다 매번 같은 순서를 밟았다 — 원인을 조사하고, 요청서(`request.md`)를 정해진 형식으로 쓰고, 결과(`result.md`)가 오면 검증하고, 완료 폴더로 옮기고, `STATUS.md`를 남긴다. 절차 자체는 이미 몸에 익었지만, 매번 처음부터 다시 설명하듯 진행하고 있었다.

이런 "매번 똑같은 순서를 밟는 일"이야말로 Skill로 만들기 딱 좋은 대상이다.

## Skill에게 시키는 게 아니라, 절차를 적어두는 것

> 지금까지 우리가 handoff를 발행하고 완료 처리하는 순서를 보면, 매번 똑같은 단계를 밟고 있어. 이걸 Skill로 만들어서, 다음부턴 "handoff 요청서 써줘"라고만 해도 이 순서 그대로 진행되게 해줘.

Skill은 새로운 능력을 추가하는 게 아니다. 이미 할 줄 아는 일을 "이 순서로, 이 형식으로 하라"고 미리 적어두는 절차서에 가깝다. 이 프로젝트의 handoff 절차로 예를 들면 이런 단계들이 들어갈 수 있다.

1. `.local/handoff/inbox/<패키지명>/request.md`를 정해진 형식(목적, 참조자료, 포함범위, 제외범위, Acceptance Criteria, 실행 문구)으로 작성한다
2. 작성 즉시 발행 완료로 간주한다(별도 확인 절차 없음)
3. `result.md`가 도착하면 내용을 검토하고, 필요하면 실제 배포 결과를 직접 재검증한다
4. `.local/handoff/completed/<패키지명>/`으로 옮기고 `STATUS.md`를 작성한다
5. Tommy의 확인이 필요한 경우 Completed로, 직접 확인까지 끝나면 Confirmed로 표시한다

## 매번 설명하지 않아도 되는 것들

Skill로 한번 정리해두면, 다음번엔 "handoff 하나 만들어줘"라는 말 한마디로 이 다섯 단계가 그대로 실행된다. 매번 "아 맞다, 완료되면 STATUS.md도 같이 써야 하지" 하고 떠올릴 필요가 없어진다. 절차가 몸에 밴 사람에게도 도움이 되지만, 특히 이 프로젝트를 처음 보는 사람(또는 새 세션을 시작한 Claudox)에게는 그 자체로 온보딩 문서 역할을 한다.

## 반복이 보일 때가 적기

Skill을 만들 타이밍을 판단하는 기준은 단순하다 — 같은 순서를 세 번 이상 손으로 반복하고 있다면, 그건 이미 절차가 확립됐다는 뜻이다. 그 시점에 "이거 Skill로 만들어줘"라고 요청하면, 그동안 실제로 밟았던 순서를 그대로 문서화해서 재사용 가능한 형태로 만들 수 있다. 처음부터 완벽한 절차를 설계하려 하기보다, **실제로 여러 번 해보고 나서 정리하는 쪽**이 훨씬 정확하다.

---

*이 장은 아직 이 프로젝트에서 실제로 Skill을 만들어본 기록이 아니라, 앞으로 직접 만들어볼 시나리오로 미리 써둔 것이다. handoff 절차는 이미 여러 번 반복된 좋은 후보이니, 실제로 Skill을 만들어보면 이 장은 진짜 있었던 이야기로 다시 쓰일 예정이다.*
