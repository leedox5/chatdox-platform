# 13. 리팩토링 실전

체크아웃을 여러 번 시도해도 결제 대기 주문이 쌓이지 않게 해달라는 요청은 간단해 보였다. "이미 대기 중인 주문이 있으면 새로 만들지 말고 재사용하라"는 조건 하나만 주문 생성 함수 안에 넣으면 될 것 같았다.

Platform Agent도 처음엔 그렇게 했다. 요청서가 가리킨 그대로, 저수준 주문 생성 함수(`OrderCreator`) 안에 중복 확인 로직을 바로 심었다.

## 테스트가 먼저 알아챘다

`bin/rails test`를 돌리자 전혀 관련 없어 보이던 기존 테스트 6개가 갑자기 실패했다. 원인을 들여다보니, 그 테스트들은 "결제 확인 순서가 뒤바뀌어도 문제없는지" 같은 시나리오를 검증하기 위해 **같은 조건으로 독립된 주문 여러 개를 일부러 만들고 있었다**. 방금 넣은 중복 방지 로직은 정확히 그 전제를 깨버렸다.

이 함수는 처음 짤 때부터 "요청한 그대로 정확히 새 주문 하나를 만드는" 낮은 층위의 도구로 설계돼 있었고, 재시도 기능이나 테스트 코드 곳곳에서 이미 그 계약을 믿고 쓰이고 있었다. 중복 방지라는 새로운 정책을 그 안에 밀어넣는 건, 여러 곳이 공유하는 도구의 원래 약속을 슬쩍 바꿔버리는 일이었다.

## 정책과 도구를 다시 나누다

해결책은 코드를 더 고치는 게 아니라, **어디에 놓을지를 바꾸는 것**이었다. 중복 방지는 실제 체크아웃 화면이 폼을 제출하는 그 진입점에만 필요한 정책이었다. 그래서 `OrderCreator`는 원래 하던 일 그대로 남겨두고, 그 앞단에 얇은 서비스(`CheckoutSubmission`) 하나를 새로 만들어 정책을 거기로 옮겼다.

```
기존:  체크아웃 화면 → OrderCreator (여기에 중복 방지까지 욱여넣음)

변경:  체크아웃 화면 → CheckoutSubmission (중복 방지 판단)
                              └─ 필요할 때만 → OrderCreator (원래 그대로)
```

`OrderCreator`를 쓰던 다른 곳(재시도 기능, 테스트 픽스처)은 전혀 건드릴 필요가 없었다. 결과적으로 최종 diff에서 `order_creator.rb` 파일 자체는 변경 없이 그대로 남았다 — 실제로 `git diff`로 그걸 확인하는 것으로 이번 리팩토링이 끝났음을 검증했다.

## 리팩토링은 실패에서 시작됐다

이번 사례가 보여주는 건, 좋은 설계가 처음부터 나오지 않아도 괜찮다는 것이었다. 처음 시도(저수준 함수 안에 정책 심기)는 자연스러운 선택이었고, 그게 틀렸다는 것도 코드 리뷰가 아니라 **테스트 실패**가 먼저 알려줬다. 중요했던 건 실패를 보고 테스트를 억지로 고쳐서 통과시키는 대신, "왜 이 테스트들이 이런 전제를 갖고 있었는지"부터 이해하고 넘어간 것이었다. 그 이해가 있었기에 "정책은 진입점에, 저수준 도구는 그대로"라는 더 나은 경계선을 찾을 수 있었다.
