# 15. Pull Request를 만들다

지금까지는 대부분 이런 흐름이었다. Claudox가 코드를 고치고, 테스트를 돌리고, 바로 main 브랜치에 커밋해서 push한다. 작은 팀, 빠른 반복에는 이 방식이 잘 맞았다. 하지만 문득 궁금해졌다 — 만약 다른 사람과 같이 코드를 보는 과정, 즉 Pull Request(PR)를 거쳐야 하는 상황이라면 이 협업은 어떻게 달라질까.

## 브랜치부터 나눈다

> 지금 이 수정 건은 바로 main에 올리지 말고, 브랜치를 따로 만들어서 PR로 올려줘. 리뷰 절차를 한번 밟아보고 싶어.

이 한마디로 흐름이 바뀐다. Claudox는 지금 작업 중인 main 브랜치 대신, 이 변경만 담을 새 브랜치를 만든다. 브랜치 이름은 보통 무엇을 하는지 알 수 있게 짓는다 — `fix/duplicate-pending-orders`처럼, 나중에 브랜치 목록만 봐도 무슨 작업이었는지 짐작할 수 있게.

```bash
git checkout -b fix/duplicate-pending-orders
```

이 브랜치 위에서 코드를 고치고, 커밋하고, 테스트를 돌리는 과정은 지금까지와 똑같다. 달라지는 건 이 브랜치를 곧바로 main에 합치지 않는다는 것뿐이다.

## PR을 연다

작업이 끝나면 이 브랜치를 원격 저장소로 올리고, PR을 연다.

```bash
git push -u origin fix/duplicate-pending-orders
gh pr create --title "체크아웃 재제출 시 pending 주문 중복 생성 방지" --body "..."
```

PR 설명에는 보통 세 가지가 들어간다 — 무엇을 바꿨는지, 왜 바꿨는지, 그리고 어떻게 검증했는지. 커밋 메시지가 "무엇을"에 집중한다면, PR 설명은 "왜"와 "이게 안전하다는 근거"까지 담아서 리뷰하는 사람이 처음부터 코드를 다 읽지 않아도 맥락을 잡을 수 있게 돕는다.

## 리뷰를 받는다

PR이 열리면 그때부터는 코드가 "일단 완성된 것"이 아니라 "검토 대상"이 된다. 리뷰하는 사람은 코드 자체의 문제뿐 아니라, "이 방식이 맞는 접근인가"까지 물어볼 수 있다. 이 프로젝트의 맥락으로 보면, 13장의 리팩토링 사례처럼 "저수준 함수에 정책을 직접 넣었다가 기존 테스트가 깨진" 문제도, PR 단계에서 리뷰어가 먼저 짚어줬을 수 있는 종류의 문제다.

> 이 PR에 달린 코멘트들을 보고 반영해줘. 특히 기존 코드 재사용 관련 코멘트는 꼼꼼히 봐줘.

리뷰 코멘트에 대한 대응도 그대로 요청하면 된다. Claudox가 코멘트를 읽고, 필요하면 같은 브랜치에 커밋을 추가하고, 다시 push한다 — PR은 자동으로 최신 커밋을 반영한다.

## 병합하고 마무리한다

리뷰가 끝나면 병합한다.

```bash
gh pr merge --squash
```

`--squash`는 브랜치에 쌓인 여러 개의 작은 커밋을 하나로 합쳐서 main에 반영한다. 작업 과정에서 생긴 "오타 수정", "테스트 추가" 같은 자잘한 커밋들이 main의 히스토리를 어지럽히지 않게 해준다.
