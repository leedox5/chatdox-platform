# 17. 워크플로우로 규모를 키우다

역할을 나눈 것만으로 팀이 움직이지는 않았다. 서로 기억을 공유하지 않는 기획자와 구현자가 같은 결과를 향하려면, 말로 한 요청을 전달 가능한 작업 계약으로 바꿔야 했다.

그 연결 장치가 handoff였다.

## 요청이 패키지가 되다

LEEDOX 통합 홈을 기획할 때 Chatdox는 구현 지시 한 줄만 넘기지 않았다. 필요한 맥락을 하나의 패키지로 구성했다.

| 파일 | 역할 |
|---|---|
| request.md | 포함 범위, 제외 범위와 Acceptance Criteria |
| product.md | 브랜드, 고객과 상품의 기획 원본 |
| guide.md | 기획·구현·검증 사이의 협업 원칙 |
| landing.png | 변경 전 화면과 시각적 기준 |

구현 계약서에는 새 홈에 무엇이 보여야 하는지만 적지 않았다. 기존 인증과 결제, 문서와 대시보드를 보존할 것, 미확정 가격을 사실처럼 표시하지 않을 것, 기존 Chatdox 홈을 삭제하지 않을 것도 명시했다.

기술 해법은 구현자가 현재 코드를 조사한 뒤 결정하도록 남겨뒀다. 기획은 목적과 경계를 정하고, 구현은 현실의 코드 안에서 방법을 선택했다.

## 첫 번째 실행

Platform Agent는 기존 홈이 일곱 개 partial로 구성되어 있다는 사실을 확인했다. 이를 버리지 않고 별도의 Chatdox 경로에서 그대로 재사용했다. 루트 경로에는 LEEDOX 상위 브랜드와 Chatdox·Claudox 두 상품을 소개하는 새 홈을 만들었다.

기존 Claudox 문서 뷰어, 인증과 결제 데이터 구조는 건드리지 않았다. 공통 Header와 Footer만 새로운 브랜드 구조에 맞춰 확장했다.

구현이 끝난 뒤에는 완료됐다는 한 문장 대신 결과 보고서가 돌아왔다.

- 조사한 기존 구조
- 생성·수정한 파일
- 기존 기능 보존 여부
- 실행한 테스트와 실제 결과
- 화면 크기별 검증
- 미결정 사항
- Acceptance Criteria별 판정
- 데스크톱·모바일 스크린샷

이 보고서 덕분에 기획자는 코드를 직접 작성하지 않아도 무엇이 바뀌었고 무엇이 아직 확인되지 않았는지 검토할 수 있었다.

## 보류가 다음 작업을 만들었다

첫 결과에서 17개 기준 중 14개는 근거와 함께 통과했다. 세 항목은 보류였다.

- LEEDOX가 상위 브랜드로 즉시 이해되는가
- 키보드만으로 주요 기능을 사용할 수 있는가
- 브라우저 Console에 새 오류가 없는가

Chatdox의 화면 검토에서는 모바일 한글 제목의 줄바꿈 문제도 발견됐다. 하지만 전체 요청서를 다시 쓰거나 R1 결과를 덮어쓰지 않았다. 필요한 변경만 R2로 좁혔다.

R2는 네 가지에 집중했다.

1. 주요 한글 제목에 단어 단위 줄바꿈 적용
2. 데스크톱·모바일 화면 재확인
3. Tab과 Enter를 이용한 키보드 순회
4. DevTools Console 오류 수집과 테스트 재실행

수정 후 전체 구조는 그대로 유지됐고, 보류됐던 항목도 증거를 갖춰 통과했다. 17개 Acceptance Criteria가 모두 충족됐다.

## 요청서 자체도 여러 번 뒤집히다

모든 handoff가 처음부터 완성된 요청서로 시작하지는 않았다. REQ 0023(DB 기반 서비스데스크)이 그런 사례였다.

시작은 단순했다 — 서비스데스크 홈 화면을 정리하고, 웹에서 티켓을 발행하고 작업 기록을 남기는 기능을 추가해달라는 요청. 그런데 "repo가 아니라 웹에서"라는 한 문장이 지금까지 HQ의 git 저장소를 유일한 진실 소스로 삼던 구조 전체를 흔들었다.

Claudox는 곧바로 구현에 들어가지 않았다. 먼저 feasibility를 검토해 의견부터 냈다 — DB를 쓰느냐 마느냐보다 진짜 걸림돌은 따로 있다고. Railway의 휘발성 파일시스템, 읽기전용 미러와의 충돌, 그리고 웹에서 만든 기록을 다시 HQ로 되돌릴 경로가 없다는 것. 세 가지였다.

Tommy는 이 의견을 그대로 받아들이는 대신 질문 자체를 뒤집었다. "그럼 아예 git 체계를 벗어나서 순수하게 웹 기반으로 가는 게 가능한가?" Claudox는 그 질문에 맞춰 트레이드오프를 다시 계산했다 — 가능하지만 Railway 프로젝트 하나가 유일한 사본이 되는 단일 장애점이 새로 생긴다고. 몇 차례 더 주고받은 끝에 Tommy는 "두 시스템이 굳이 동기화될 필요는 없다, git 쪽은 용도를 따로 정의해서 계속 쓰면 된다"는 결정으로 정리했다.

이 결정 하나로 처음에 가장 어려워 보였던 두 단계 — AI 에이전트 인증, HQ로의 역방향 동기화 — 가 통째로 사라졌다. 견적은 절반 가까이 줄었다.

마지막 협상거리는 기존 23건 요청 중 무엇을 새 시스템으로 옮길지였다. Claudox가 기준(git은 정책·구조 결정 기록, DB는 웹 기반 운영 로그)을 제안하고 14건과 9건으로 나눴는데, "구현"이라는 같은 동사를 쓴 두 티켓이 서로 다른 쪽에 배정된 지점은 판단이 갈릴 수 있다고 미리 짚어뒀다. Tommy는 그 제안을 그대로 확정했다 — 빈 시스템보다는 실제 히스토리를 갖춘 채로 시작하는 편이 낫다는 판단이었다.

요청서가 handoff 패키지로 굳어진 건 이 모든 되돌림을 거친 뒤였다. 규모를 키운다는 건 첫 요청을 그대로 받아 적는 게 아니라, 구현으로 넘기기 전까지 몇 번이고 다시 물어볼 여지를 남겨두는 것이었다.

## 교육 문서화까지가 한 사이클

구현과 승인이 끝난 뒤에도 작업은 끝나지 않았다. 전체 handoff 원본은 내부 기획과 중간 스크린샷을 포함했으므로 .local에 완료 증거로 보관했다. 실제 코드와 테스트는 Platform의 공식 Git 이력이 됐다. 장기적으로 필요한 결정과 교훈은 Curriculum의 사례 문서로 정제했다.

그리고 마지막 단계가 남았다. 실제 최종 코드를 교육 자료로 정확하게 설명하려면 구현자가 기술 근거를 제공하고, Chatdox가 이를 학습 순서로 다시 편집해야 했다.

전체 흐름은 이렇게 완성됐다.

    Product Owner가 방향을 결정한다
                ↓
    Chatdox가 기획과 완료 기준을 작성한다
                ↓
    Platform Agent가 조사·구현·테스트한다
                ↓
    Chatdox가 결과와 화면을 교차 검토한다
                ↓
    R2로 필요한 부분만 보완한다
                ↓
    Product Owner가 최종 승인한다
                ↓
    코드·로컬 증거·교육 지식을 분리한다

규모를 키운다는 것은 요청을 많이 던지는 일이 아니었다. 각 단계의 입력과 출력, 책임자와 완료 기준을 분명히 해서 다음 팀원이 이어받을 수 있게 만드는 일이었다.
