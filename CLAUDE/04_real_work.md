# 4. 본격적인 작업 — 생산성이 달라지다

이름을 정하고 규칙을 세우고 나니, 슬슬 진짜 일거리가 생기기 시작했다. 처음엔 채팅으로 "이거 해줘" 하고 던지는 식이었는데, 요청이 쌓이다 보니 뭘 요청했는지, 뭐가 끝났는지 헷갈리기 시작했다. 그래서 요청 자체를 관리하는 시스템이 필요해졌다.

## 서비스데스크, 요청이 일이 되는 곳

처음엔 `CLAUDE/99_service_desk.md`라는 파일 하나에 `REQ_NNNN>` 형식으로 요청을 적어두는 정도였다. 그러다 이걸로는 부족하다 싶어, 아예 리포 루트에 `service-desk/` 폴더를 만들고 제대로 된 워크플로우로 발전시켰다.

- `01_new/` → `02_in_progress/` → `03_completed/`: 요청 파일이 처리 상태에 따라 폴더를 옮겨 다닌다.
- `_FORM.md`는 언제든 복사해서 쓰는 빈 양식이고, `new.sh`/`new.ps1`은 이 양식을 복사해 다음 ID를 자동으로 매겨준다.
- 완료된 요청은 지우지 않고 `03_completed/`에 `Job` 내역과 함께 남긴다 — 뭘 했는지가 기록으로 쌓인다.

여기서 한 가지 선을 그었다. service-desk는 채팅에서 나온 모든 이야기를 자동으로 담는 곳이 아니다. `CLAUDE/` 챕터처럼 "모든 대화를 기록한다"는 규칙과는 다르게, **실제로 접수된 요청만** 다룬다.

## 실제로 처리된 것들

이 시스템이 굴러가기 시작하면서, 짧은 시간에 벌써 몇 건이 처리됐다.

- **REQ 0001** — 1장에 Free/Pro/Max 요금제 비교를 추가해달라는 요청.
- **REQ 0002** — service-desk 자체를 살펴보고 완성도를 높여달라는, 스스로를 다듬는 자기참조적 요청.
- **REQ 0003** — `docs/03_dev_setup.md`에 WSL 환경에서의 Ruby 설치 방법을 추가해달라는 요청.

요청 하나하나는 작았지만, "요청 → 처리 → 기록"이 매번 같은 방식으로 굴러가기 시작하자 확실히 달라진 게 느껴졌다. 예전엔 대화 속에 묻혀 있던 요청들이, 이제는 빠짐없이 시작되고 끝나는 하나의 흐름이 되었다.
